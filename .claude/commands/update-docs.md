# Update Documentation

소스 진실 파일에서 생성하여 코드베이스와 문서를 동기화합니다.

## 1단계: 진실의 원천 식별

| 소스 | 생성 내용 |
|------|-----------|
| `package.json` 스크립트 | 사용 가능한 명령어 참조 |
| `.env.example` | 환경 변수 문서 |
| `openapi.yaml` / 라우트 파일 | API 엔드포인트 참조 |
| 소스 코드 exports | 공개 API 문서 |
| `Dockerfile` / `docker-compose.yml` | 인프라 설정 문서 |

## 2단계: 스크립트 참조 생성

1. `package.json` (또는 `Makefile`, `Cargo.toml`, `pyproject.toml`) 읽기
2. 설명과 함께 모든 스크립트/명령어 추출
3. 참조 테이블 생성:

```markdown
| 명령어 | 설명 |
|--------|------|
| `npm run dev` | 핫 리로드로 개발 서버 시작 |
| `npm run build` | 타입 검사를 포함한 프로덕션 빌드 |
| `npm test` | 커버리지를 포함한 테스트 스위트 실행 |
```

## 3단계: 환경 문서 생성

1. `.env.example` (또는 `.env.template`, `.env.sample`) 읽기
2. 목적과 함께 모든 변수 추출
3. 필수 vs 선택 분류
4. 예상 형식 및 유효한 값 문서화

```markdown
| 변수 | 필수 | 설명 | 예시 |
|------|------|------|------|
| `DATABASE_URL` | 예 | PostgreSQL 연결 문자열 | `postgres://user:pass@host:5432/db` |
| `LOG_LEVEL` | 아니오 | 로깅 상세도 (기본값: info) | `debug`, `info`, `warn`, `error` |
```

## 4단계: 기여 가이드 업데이트

다음 내용으로 `docs/CONTRIBUTING.md` 생성 또는 업데이트:
- 개발 환경 설정 (전제 조건, 설치 단계)
- 사용 가능한 스크립트와 목적
- 테스트 절차 (실행 방법, 새 테스트 작성 방법)
- 코드 스타일 적용 (린터, 포매터, pre-commit 훅)
- PR 제출 체크리스트

## 5단계: 런북 업데이트

다음 내용으로 `docs/RUNBOOK.md` 생성 또는 업데이트:

## 규칙

- **단일 진실의 원천**: 항상 코드에서 생성하고, 생성된 섹션을 수동으로 편집하지 마세요
- **수동 섹션 보존**: 생성된 섹션만 업데이트하고 수작업으로 작성된 내용은 유지
- **생성된 내용 표시**: 생성된 섹션 주변에 `<!-- AUTO-GENERATED -->` 마커 사용
- **명시적 요청 없이 문서 생성 금지**: 명령이 명시적으로 요청하는 경우에만 새 문서 파일 생성
