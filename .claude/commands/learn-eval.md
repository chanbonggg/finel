# /learn-eval - 추출, 평가 후 저장

저장 전 품질 게이트와 저장 위치 결정을 추가하여 `/learn`을 확장합니다.

## 추출 대상

다음을 찾습니다:

1. **에러 해결 패턴** — 근본 원인 + 수정 + 재사용성
2. **디버깅 기법** — 비직관적인 단계, 도구 조합
3. **우회 방법** — 라이브러리 특이 동작, API 제한, 버전별 수정
4. **프로젝트별 패턴** — 관례, 아키텍처 결정, 통합 패턴

## 프로세스

1. 추출 가능한 패턴이 있는지 세션 검토
2. 가장 가치 있고 재사용 가능한 인사이트 식별

3. **저장 위치 결정:**
   - 질문: "이 패턴이 다른 프로젝트에서도 유용할까요?"
   - **전역** (`~/.claude/skills/learned/`): 2개 이상의 프로젝트에서 사용 가능한 범용 패턴 (bash 호환성, LLM API 동작, 디버깅 기법 등)
   - **프로젝트** (현재 프로젝트의 `.claude/skills/learned/`): 프로젝트별 지식 (특정 설정 파일의 특이 동작, 프로젝트 특화 아키텍처 결정 등)
   - 불확실할 때는 전역 선택 (전역 → 프로젝트 이동이 반대보다 쉬움)

4. 다음 형식으로 스킬 파일 초안 작성:

```markdown
---
name: pattern-name
description: "130자 이내"
user-invocable: false
origin: auto-extracted
---

# [설명적인 패턴 이름]

**추출일:** [날짜]
**컨텍스트:** [적용 시점에 대한 간략한 설명]

## 문제
[이 패턴이 해결하는 문제 - 구체적으로]

## 해결책
[패턴/기법/우회 방법 - 코드 예시 포함]

## 사용 시점
[트리거 조건]
```

5. **저장 전 다음 기준으로 자체 평가:**

   | 항목 | 1 | 3 | 5 |
   |------|---|---|---|
   | 구체성 | 추상적인 원칙만, 코드 예시 없음 | 대표적인 코드 예시 있음 | 모든 사용 패턴을 포괄하는 풍부한 예시 |
   | 실행 가능성 | 수행 방법 불명확 | 주요 단계 이해 가능 | 즉시 실행 가능, 엣지 케이스 포함 |
   | 범위 적합성 | 너무 넓거나 좁음 | 대체로 적절, 경계 모호함 일부 있음 | 이름, 트리거, 내용 완벽히 일치 |
   | 비중복성 | 다른 스킬과 거의 동일 | 일부 중복되나 고유한 관점 있음 | 완전히 고유한 가치 |
   | 커버리지 | 대상 작업의 일부만 커버 | 주요 케이스 커버, 일반 변형 누락 | 주요 케이스, 엣지 케이스, 함정 모두 커버 |

   - 각 항목 1-5점 채점
   - 1-2점인 항목이 있으면 초안 개선 후 재채점 (모든 항목 3점 이상이 될 때까지)
   - 사용자에게 점수표와 최종 초안 표시

6. 사용자 확인 요청:
   - 표시: 제안된 저장 경로 + 점수표 + 최종 초안
   - 명시적 확인 후 작성

7. 결정된 위치에 저장

## 5단계 출력 형식 (점수표)

| 항목 | 점수 | 근거 |
|------|------|------|
| 구체성 | N/5 | ... |
| 실행 가능성 | N/5 | ... |
| 범위 적합성 | N/5 | ... |
| 비중복성 | N/5 | ... |
| 커버리지 | N/5 | ... |
| **합계** | **N/25** | |

## 참고

- 사소한 수정 (오타, 단순 문법 오류)은 추출하지 않습니다
- 일회성 문제 (특정 API 장애 등)는 추출하지 않습니다
- 미래 세션에서 시간을 절약할 패턴에 집중합니다
- 스킬은 집중적으로 유지합니다 — 스킬당 하나의 패턴
- 커버리지 점수가 낮으면 저장 전 관련 변형을 추가하세요
