# 자율 루프 스킬

Claude Code를 루프로 자율 실행하기 위한 패턴, 아키텍처, 참조 구현. 단순한 `claude -p` 파이프라인부터 완전한 RFC 기반 멀티 에이전트 DAG 오케스트레이션까지 다룹니다.

## 사용 시점

- 인간 개입 없이 실행되는 자율 개발 워크플로우 설정 시
- 문제에 맞는 루프 아키텍처 선택 시 (단순 vs 복잡)
- CI/CD 스타일 지속적 개발 파이프라인 구축 시
- 병합 조정이 있는 병렬 에이전트 실행 시
- 루프 반복 간 컨텍스트 유지 구현 시
- 자율 워크플로우에 품질 게이트 및 정리 패스 추가 시

## 루프 패턴 스펙트럼

가장 단순한 것부터 가장 정교한 것까지:

| 패턴 | 복잡도 | 최적 사용 사례 |
|------|--------|--------------|
| [순차 파이프라인](#1-순차-파이프라인-claude--p) | 낮음 | 일일 개발 단계, 스크립트 워크플로우 |
| [NanoClaw REPL](#2-nanoclaw-repl) | 낮음 | 대화형 지속 세션 |
| [무한 에이전트 루프](#3-무한-에이전트-루프) | 중간 | 병렬 콘텐츠 생성, 명세 기반 작업 |
| [지속적 Claude PR 루프](#4-지속적-claude-pr-루프) | 중간 | CI 게이트가 있는 다일 반복 프로젝트 |
| [De-Sloppify 패턴](#5-de-sloppify-패턴) | 부가 기능 | 구현 단계 후 품질 정리 |
| [Ralphinho / RFC 기반 DAG](#6-ralphinho--rfc-기반-dag-오케스트레이션) | 높음 | 대규모 기능, 병합 큐가 있는 멀티 유닛 병렬 작업 |

## 1. 순차 파이프라인 (`claude -p`)

**가장 단순한 루프.** 일일 개발을 일련의 비대화형 `claude -p` 호출로 분리합니다. 각 호출은 명확한 프롬프트를 가진 집중된 단계입니다.

### 핵심 통찰

> 이런 루프를 파악하지 못한다면, 대화형 모드에서도 LLM으로 코드를 수정하도록 유도하지 못한다는 의미입니다.

`claude -p` 플래그는 프롬프트와 함께 Claude Code를 비대화형으로 실행하고, 완료 시 종료합니다. 호출을 연결하여 파이프라인을 구축합니다:

```bash
#!/bin/bash
# daily-dev.sh — 기능 브랜치를 위한 순차 파이프라인

set -e

# 1단계: 기능 구현
claude -p "docs/auth-spec.md의 명세를 읽고 src/auth/에 OAuth2 로그인을 구현하세요. 테스트를 먼저 작성하세요 (TDD). 새로운 문서 파일을 생성하지 마세요."

# 2단계: De-sloppify (정리 패스)
claude -p "이전 커밋에서 변경된 모든 파일을 검토하세요. 불필요한 타입 테스트, 과도하게 방어적인 검사, 또는 언어 기능 테스트(예: TypeScript 제네릭 작동 테스트)를 제거하세요. 실제 비즈니스 로직 테스트는 유지하세요. 정리 후 테스트 스위트를 실행하세요."

# 3단계: 검증
claude -p "전체 빌드, 린트, 타입 검사, 테스트 스위트를 실행하세요. 실패를 수정하세요. 새 기능을 추가하지 마세요."

# 4단계: 커밋
claude -p "스테이징된 모든 변경사항에 대해 관례적 커밋을 생성하세요. 'feat: add OAuth2 login flow'를 메시지로 사용하세요."
```

### 핵심 설계 원칙

1. **각 단계는 격리됨** — `claude -p` 호출마다 새 컨텍스트 창이 시작되어 단계 간 컨텍스트 오염이 없음.
2. **순서가 중요** — 단계는 순차적으로 실행됩니다. 각 단계는 이전 단계가 파일 시스템에 남긴 상태 위에 구축됩니다.
3. **부정적 지시는 위험** — "타입 시스템을 테스트하지 마세요"라고 말하지 마세요. 대신 별도의 정리 단계를 추가하세요 ([De-Sloppify 패턴](#5-de-sloppify-패턴) 참조).
4. **종료 코드 전파** — `set -e`는 실패 시 파이프라인을 중지합니다.

### 변형

**모델 라우팅 사용:**
```bash
# 리서치에 Opus 사용 (심층 추론)
claude -p --model opus "코드베이스 아키텍처를 분석하고 캐싱 추가 계획을 작성하세요..."

# 구현에 Sonnet 사용 (빠르고 유능함)
claude -p "docs/caching-plan.md의 계획에 따라 캐싱 레이어를 구현하세요..."

# 검토에 Opus 사용 (철저함)
claude -p --model opus "보안 문제, 경쟁 조건, 엣지 케이스에 대해 모든 변경사항을 검토하세요..."
```

**환경 컨텍스트 사용:**
```bash
# 프롬프트 길이가 아닌 파일을 통해 컨텍스트 전달
echo "Focus areas: auth module, API rate limiting" > .claude-context.md
claude -p ".claude-context.md에서 우선순위를 읽으세요. 순서대로 작업하세요."
rm .claude-context.md
```

**`--allowedTools` 제한 사용:**
```bash
# 읽기 전용 분석 패스
claude -p --allowedTools "Read,Grep,Glob" "보안 취약점에 대해 이 코드베이스를 감사하세요..."

# 쓰기 전용 구현 패스
claude -p --allowedTools "Read,Write,Edit,Bash" "security-audit.md의 수정사항을 구현하세요..."
```

## 2. NanoClaw REPL

**ECC의 내장 지속 루프.** 전체 대화 기록과 함께 `claude -p`를 동기적으로 호출하는 세션 인식 REPL.

```bash
# 기본 세션 시작
node scripts/claw.js

# 스킬 컨텍스트가 있는 이름 지정 세션
CLAW_SESSION=my-project CLAW_SKILLS=tdd-workflow,security-review node scripts/claw.js
```

### 작동 방식

1. `~/.claude/claw/{session}.md`에서 대화 기록 로드
2. 각 사용자 메시지는 전체 기록을 컨텍스트로 `claude -p`에 전송
3. 응답이 세션 파일에 추가됨 (데이터베이스로서의 Markdown)
4. 세션은 재시작 후에도 유지됨

### NanoClaw vs 순차 파이프라인 선택

| 사용 사례 | NanoClaw | 순차 파이프라인 |
|----------|----------|--------------|
| 대화형 탐색 | 예 | 아니오 |
| 스크립트 자동화 | 아니오 | 예 |
| 세션 유지 | 내장 | 수동 |
| 컨텍스트 누적 | 턴마다 증가 | 단계마다 초기화 |
| CI/CD 통합 | 나쁨 | 우수 |

자세한 내용은 `/claw` 명령어 문서를 참조하세요.

## 3. 무한 에이전트 루프

**두 개의 프롬프트 시스템**으로 명세 기반 생성을 위한 병렬 하위 에이전트를 오케스트레이션합니다. disler(credit: @disler)가 개발했습니다.

### 아키텍처: 두 프롬프트 시스템

```
프롬프트 1 (오케스트레이터)          프롬프트 2 (하위 에이전트)
┌─────────────────────┐             ┌──────────────────────┐
│ 명세 파일 파싱        │             │ 전체 컨텍스트 수신     │
│ 출력 디렉토리 스캔    │  배포       │ 할당된 번호 읽기       │
│ 반복 계획            │────────────│ 명세 정확히 따르기     │
│ 창의적 방향 할당      │  N 에이전트 │ 고유 출력 생성         │
│ 웨이브 관리          │             │ 출력 디렉토리에 저장   │
└─────────────────────┘             └──────────────────────┘
```

### 패턴

1. **명세 분석** — 오케스트레이터가 생성할 내용을 정의한 명세 파일(Markdown)을 읽음
2. **디렉토리 정찰** — 기존 출력을 스캔하여 최고 반복 번호 찾기
3. **병렬 배포** — N개의 하위 에이전트를 각각 다음과 함께 실행:
   - 전체 명세
   - 고유한 창의적 방향
   - 특정 반복 번호 (충돌 없음)
   - 기존 반복의 스냅샷 (고유성을 위해)
4. **웨이브 관리** — 무한 모드에서, 컨텍스트가 소진될 때까지 3-5개 에이전트의 웨이브 배포

### Claude Code 명령어를 통한 구현

`.claude/commands/infinite.md` 생성:

```markdown
$ARGUMENTS에서 다음 인수를 파싱:
1. spec_file — 명세 마크다운 경로
2. output_dir — 반복이 저장되는 위치
3. count — 정수 1-N 또는 "infinite"

단계 1: 명세를 읽고 깊이 이해합니다.
단계 2: output_dir을 나열하고, 가장 높은 반복 번호를 찾습니다. N+1에서 시작합니다.
단계 3: 창의적 방향을 계획합니다 — 각 에이전트는 다른 테마/접근 방식을 받습니다.
단계 4: 하위 에이전트를 병렬로 배포합니다 (Task 도구). 각각 다음을 받습니다:
  - 전체 명세 텍스트
  - 현재 디렉토리 스냅샷
  - 할당된 반복 번호
  - 고유한 창의적 방향
단계 5 (무한 모드): 컨텍스트가 부족할 때까지 3-5의 웨이브로 루프합니다.
```

**실행:**
```bash
/project:infinite specs/component-spec.md src/ 5
/project:infinite specs/component-spec.md src/ infinite
```

### 배칭 전략

| 수량 | 전략 |
|------|------|
| 1-5 | 모든 에이전트 동시 실행 |
| 6-20 | 5개씩 배치 처리 |
| infinite | 3-5의 웨이브, 점진적 복잡도 증가 |

### 핵심 통찰: 할당을 통한 고유성

에이전트가 스스로 차별화하도록 의존하지 마세요. 오케스트레이터가 각 에이전트에게 특정 창의적 방향과 반복 번호를 **할당**합니다. 이를 통해 병렬 에이전트 간의 중복 개념을 방지합니다.

## 4. 지속적 Claude PR 루프

**프로덕션 수준 쉘 스크립트**로 Claude Code를 지속적 루프로 실행하고, PR을 생성하며, CI를 기다리고, 자동으로 병합합니다. AnandChowdhary(credit: @AnandChowdhary)가 만들었습니다.

### 핵심 루프

```
┌─────────────────────────────────────────────────────┐
│  지속적 CLAUDE 반복                                  │
│                                                     │
│  1. 브랜치 생성 (continuous-claude/iteration-N)      │
│  2. 향상된 프롬프트로 claude -p 실행                  │
│  3. (선택) 검토자 패스 — 별도의 claude -p            │
│  4. 변경사항 커밋 (claude가 메시지 생성)               │
│  5. 푸시 + PR 생성 (gh pr create)                   │
│  6. CI 검사 대기 (gh pr checks 폴링)                 │
│  7. CI 실패? → 자동 수정 패스 (claude -p)            │
│  8. PR 병합 (squash/merge/rebase)                   │
│  9. main으로 돌아가기 → 반복                         │
│                                                     │
│  제한: --max-runs N | --max-cost $X                  │
│         --max-duration 2h | 완료 신호                │
└─────────────────────────────────────────────────────┘
```

### 설치

```bash
curl -fsSL https://raw.githubusercontent.com/AnandChowdhary/continuous-claude/HEAD/install.sh | bash
```

### 사용법

```bash
# 기본: 10회 반복
continuous-claude --prompt "테스트되지 않은 모든 함수에 단위 테스트 추가" --max-runs 10

# 비용 제한
continuous-claude --prompt "모든 린터 오류 수정" --max-cost 5.00

# 시간 제한
continuous-claude --prompt "테스트 커버리지 개선" --max-duration 8h

# 코드 검토 패스 포함
continuous-claude \
  --prompt "인증 기능 추가" \
  --max-runs 10 \
  --review-prompt "npm test && npm run lint 실행, 실패 수정"

# 워크트리를 통한 병렬 실행
continuous-claude --prompt "테스트 추가" --max-runs 5 --worktree tests-worker &
continuous-claude --prompt "코드 리팩토링" --max-runs 5 --worktree refactor-worker &
wait
```

### 반복 간 컨텍스트: SHARED_TASK_NOTES.md

핵심 혁신: `SHARED_TASK_NOTES.md` 파일이 반복 간에 지속됩니다:

```markdown
## 진행 상황
- [x] auth 모듈에 테스트 추가 (반복 1)
- [x] 토큰 갱신의 엣지 케이스 수정 (반복 2)
- [ ] 아직 필요: 속도 제한 테스트, 오류 경계 테스트

## 다음 단계
- 다음은 속도 제한 모듈에 집중
- tests/helpers.ts의 목 설정을 재사용할 수 있음
```

Claude는 반복 시작 시 이 파일을 읽고 반복 종료 시 업데이트합니다. 이를 통해 독립적인 `claude -p` 호출 간의 컨텍스트 격차를 해소합니다.

### CI 실패 복구

PR 검사가 실패하면, Continuous Claude가 자동으로:
1. `gh run list`를 통해 실패한 실행 ID 가져오기
2. CI 수정 컨텍스트로 새 `claude -p` 실행
3. Claude가 `gh run view`를 통해 로그를 검사하고, 코드를 수정하고, 커밋하고, 푸시
4. 검사를 다시 대기 (최대 `--ci-retry-max` 시도 횟수)

### 완료 신호

Claude는 마법 구문을 출력하여 "완료"를 신호할 수 있습니다:

```bash
continuous-claude \
  --prompt "이슈 트래커의 모든 버그 수정" \
  --completion-signal "CONTINUOUS_CLAUDE_PROJECT_COMPLETE" \
  --completion-threshold 3  # 3번 연속 신호 후 중지
```

완료를 신호하는 3번의 연속 반복이 루프를 중지하여, 완료된 작업에 낭비되는 실행을 방지합니다.

### 주요 구성

| 플래그 | 목적 |
|--------|------|
| `--max-runs N` | N회 성공적인 반복 후 중지 |
| `--max-cost $X` | $X 지출 후 중지 |
| `--max-duration 2h` | 시간 경과 후 중지 |
| `--merge-strategy squash` | squash, merge, 또는 rebase |
| `--worktree <name>` | git 워크트리를 통한 병렬 실행 |
| `--disable-commits` | 드라이런 모드 (git 작업 없음) |
| `--review-prompt "..."` | 반복당 검토자 패스 추가 |
| `--ci-retry-max N` | CI 실패 자동 수정 (기본값: 1) |

## 5. De-Sloppify 패턴

**모든 루프에 대한 부가 기능 패턴.** 각 구현 단계 후에 전용 정리/리팩토링 단계를 추가합니다.

### 문제

LLM에게 TDD로 구현을 요청할 때 "테스트 작성"을 너무 문자 그대로 받아들입니다:
- TypeScript 타입 시스템 작동 검증 테스트 (예: `typeof x === 'string'` 테스트)
- 타입 시스템이 이미 보장하는 것에 대한 과도하게 방어적인 런타임 검사
- 비즈니스 로직이 아닌 프레임워크 동작에 대한 테스트
- 실제 코드를 가리는 과도한 오류 처리

### 부정적 지시가 안 되는 이유

구현자 프롬프트에 "타입 시스템을 테스트하지 마세요" 또는 "불필요한 검사를 추가하지 마세요"를 추가하면 다운스트림 효과가 있습니다:
- 모델이 모든 테스트에 대해 주저하게 됨
- 합법적인 엣지 케이스 테스트를 건너뜀
- 품질이 예측 불가능하게 저하됨

### 해결책: 별도 패스

구현자를 제한하는 대신, 철저하게 하도록 허용합니다. 그런 다음 집중된 정리 에이전트를 추가합니다:

```bash
# 1단계: 구현 (철저하게 허용)
claude -p "전체 TDD로 기능을 구현합니다. 테스트에서 철저하게 하세요."

# 2단계: De-sloppify (별도 컨텍스트, 집중된 정리)
claude -p "작업 트리의 모든 변경사항을 검토합니다. 다음을 제거하세요:
- 비즈니스 로직이 아닌 언어/프레임워크 동작을 검증하는 테스트
- 타입 시스템이 이미 강제하는 중복 타입 검사
- 불가능한 상태에 대한 과도하게 방어적인 오류 처리
- Console.log 구문
- 주석 처리된 코드

모든 비즈니스 로직 테스트를 유지하세요. 정리 후 테스트 스위트를 실행하여 아무것도 깨지지 않았는지 확인하세요."
```

### 루프 컨텍스트에서

```bash
for feature in "${features[@]}"; do
  # 구현
  claude -p "TDD로 $feature 구현."

  # De-sloppify
  claude -p "정리 패스: 변경사항 검토, 테스트/코드 슬롭 제거, 테스트 실행."

  # 검증
  claude -p "빌드 + 린트 + 테스트 실행. 실패 수정."

  # 커밋
  claude -p "메시지와 함께 커밋: feat: add $feature"
done
```

### 핵심 통찰

> 다운스트림 품질 효과가 있는 부정적 지시를 추가하는 대신, 별도의 de-sloppify 패스를 추가하세요. 두 개의 집중된 에이전트가 하나의 제한된 에이전트보다 나은 성과를 냅니다.

## 6. Ralphinho / RFC 기반 DAG 오케스트레이션

**가장 정교한 패턴.** RFC 기반, 멀티 에이전트 파이프라인으로 명세를 의존성 DAG로 분해하고, 각 유닛을 계층적 품질 파이프라인을 통해 실행하고, 에이전트 기반 병합 큐를 통해 착지시킵니다. enitrat(credit: @enitrat)이 만들었습니다.

### 아키텍처 개요

```
RFC/PRD 문서
       │
       ▼
  분해 (AI)
  RFC를 의존성 DAG가 있는 작업 유닛으로 분리
       │
       ▼
┌──────────────────────────────────────────────────────┐
│  RALPH 루프 (최대 3 패스)                             │
│                                                      │
│  각 DAG 레이어마다 (순차적, 의존성 순서):               │
│                                                      │
│  ┌── 품질 파이프라인 (유닛당 병렬) ───────────────────┐  │
│  │  각 유닛은 자체 워크트리에서:                      │  │
│  │  리서치 → 계획 → 구현 → 테스트 → 검토              │  │
│  │  (복잡도 계층에 따라 깊이 변동)                   │  │
│  └──────────────────────────────────────────────────┘  │
│                                                      │
│  ┌── 병합 큐 ──────────────────────────────────────┐  │
│  │  main에 리베이스 → 테스트 실행 → 착지 또는 퇴출 │  │
│  │  퇴출된 유닛은 충돌 컨텍스트와 함께 재진입       │  │
│  └──────────────────────────────────────────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### RFC 분해

AI가 RFC를 읽고 작업 유닛을 생성합니다:

```typescript
interface WorkUnit {
  id: string;              // kebab-case 식별자
  name: string;            // 사람이 읽을 수 있는 이름
  rfcSections: string[];   // 이 유닛이 다루는 RFC 섹션
  description: string;     // 상세 설명
  deps: string[];          // 의존성 (다른 유닛 ID)
  acceptance: string[];    // 구체적인 승인 기준
  tier: "trivial" | "small" | "medium" | "large";
}
```

**분해 규칙:**
- 적고 응집력 있는 유닛 선호 (병합 위험 최소화)
- 유닛 간 파일 중복 최소화 (충돌 방지)
- 테스트는 구현과 함께 유지 ("X 구현" + "X 테스트"를 절대 분리하지 않음)
- 실제 코드 의존성이 있는 곳에만 의존성 설정

의존성 DAG가 실행 순서를 결정합니다:
```
레이어 0: [unit-a, unit-b]     ← 의존성 없음, 병렬 실행
레이어 1: [unit-c]             ← unit-a에 의존
레이어 2: [unit-d, unit-e]     ← unit-c에 의존
```

### 복잡도 계층

다른 계층은 다른 파이프라인 깊이를 받습니다:

| 계층 | 파이프라인 단계 |
|------|--------------|
| **trivial** | 구현 → 테스트 |
| **small** | 구현 → 테스트 → 코드 검토 |
| **medium** | 리서치 → 계획 → 구현 → 테스트 → PRD 검토 + 코드 검토 → 검토 수정 |
| **large** | 리서치 → 계획 → 구현 → 테스트 → PRD 검토 + 코드 검토 → 검토 수정 → 최종 검토 |

이를 통해 단순한 변경에 대한 비용이 많이 드는 작업을 방지하는 동시에, 아키텍처 변경은 철저한 검토를 받습니다.

### 별도 컨텍스트 창 (작성자 편향 제거)

각 단계는 자체 컨텍스트 창을 가진 자체 에이전트 프로세스에서 실행됩니다:

| 단계 | 모델 | 목적 |
|------|------|------|
| 리서치 | Sonnet | 코드베이스 + RFC 읽기, 컨텍스트 문서 생성 |
| 계획 | Opus | 구현 단계 설계 |
| 구현 | Codex | 계획을 따라 코드 작성 |
| 테스트 | Sonnet | 빌드 + 테스트 스위트 실행 |
| PRD 검토 | Sonnet | 명세 준수 확인 |
| 코드 검토 | Opus | 품질 + 보안 확인 |
| 검토 수정 | Codex | 검토 문제 해결 |
| 최종 검토 | Opus | 품질 게이트 (large 계층만) |

**중요한 설계:** 검토자는 검토하는 코드를 작성하지 않았습니다. 이는 자기 검토에서 놓친 문제의 가장 일반적인 원인인 작성자 편향을 제거합니다.

### 퇴출이 있는 병합 큐

품질 파이프라인이 완료된 후, 유닛은 병합 큐에 진입합니다:

```
유닛 브랜치
    │
    ├─ main에 리베이스
    │   └─ 충돌? → 퇴출 (충돌 컨텍스트 캡처)
    │
    ├─ 빌드 + 테스트 실행
    │   └─ 실패? → 퇴출 (테스트 출력 캡처)
    │
    └─ 통과 → main fast-forward, 푸시, 브랜치 삭제
```

**파일 중복 지능:**
- 비중복 유닛은 투기적으로 병렬 착지
- 중복 유닛은 매번 리베이스하면서 하나씩 착지

**퇴출 복구:**
퇴출 시, 전체 컨텍스트(충돌 파일, diff, 테스트 출력)가 캡처되어 다음 Ralph 패스에서 구현자에게 전달됩니다:

```markdown
## 병합 충돌 — 다음 착지 전 해결

이전 구현이 먼저 착지한 다른 유닛과 충돌했습니다.
아래 충돌 파일/라인을 피하도록 변경사항을 재구성하세요.

{전체 퇴출 컨텍스트와 diff}
```

### 단계 간 데이터 흐름

```
research.contextFilePath ──────────────────→ plan
plan.implementationSteps ──────────────────→ implement
implement.{filesCreated, whatWasDone} ─────→ test, reviews
test.failingSummary ───────────────────────→ reviews, implement (다음 패스)
reviews.{feedback, issues} ────────────────→ review-fix → implement (다음 패스)
final-review.reasoning ────────────────────→ implement (다음 패스)
evictionContext ───────────────────────────→ implement (병합 충돌 후)
```

### 워크트리 격리

모든 유닛은 격리된 워크트리에서 실행됩니다 (jj/Jujutsu 사용, git 아님):
```
/tmp/workflow-wt-{unit-id}/
```

같은 유닛의 파이프라인 단계는 워크트리를 **공유**하여, 리서치 → 계획 → 구현 → 테스트 → 검토 전반에 걸쳐 상태(컨텍스트 파일, 계획 파일, 코드 변경)를 유지합니다.

### 핵심 설계 원칙

1. **결정론적 실행** — 선행 분해로 병렬성과 순서를 고정
2. **레버리지 포인트에서 인간 검토** — 작업 계획이 단일 최고 레버리지 개입 지점
3. **관심사 분리** — 각 단계가 별도 컨텍스트 창과 별도 에이전트에서
4. **컨텍스트가 있는 충돌 복구** — 전체 퇴출 컨텍스트로 맹목적인 재시도가 아닌 지능적인 재실행 가능
5. **계층 기반 깊이** — 사소한 변경은 리서치/검토 건너뜀; 대규모 변경은 최대 검토
6. **재개 가능한 워크플로우** — 전체 상태가 SQLite에 유지; 어느 지점에서든 재개 가능

### Ralphinho vs 단순 패턴 선택

| 신호 | Ralphinho 사용 | 단순 패턴 사용 |
|------|--------------|--------------|
| 여러 상호 의존적 작업 유닛 | 예 | 아니오 |
| 병렬 구현 필요 | 예 | 아니오 |
| 병합 충돌 가능성 | 예 | 아니오 (순차적이면 괜찮음) |
| 단일 파일 변경 | 아니오 | 예 (순차 파이프라인) |
| 다일 프로젝트 | 예 | 아마도 (continuous-claude) |
| 명세/RFC 이미 작성됨 | 예 | 아마도 |
| 한 가지에 대한 빠른 반복 | 아니오 | 예 (NanoClaw 또는 파이프라인) |

## 올바른 패턴 선택

### 결정 매트릭스

```
작업이 단일 집중 변경인가?
├─ 예 → 순차 파이프라인 또는 NanoClaw
└─ 아니오 → 작성된 명세/RFC가 있는가?
         ├─ 예 → 병렬 구현이 필요한가?
         │        ├─ 예 → Ralphinho (DAG 오케스트레이션)
         │        └─ 아니오 → Continuous Claude (반복적 PR 루프)
         └─ 아니오 → 같은 것의 많은 변형이 필요한가?
                  ├─ 예 → 무한 에이전트 루프 (명세 기반 생성)
                  └─ 아니오 → de-sloppify가 있는 순차 파이프라인
```

### 패턴 조합

이 패턴들은 잘 조합됩니다:

1. **순차 파이프라인 + De-Sloppify** — 가장 일반적인 조합. 모든 구현 단계가 정리 패스를 받습니다.

2. **Continuous Claude + De-Sloppify** — 각 반복에 de-sloppify 지시가 있는 `--review-prompt` 추가.

3. **모든 루프 + 검증** — ECC의 `/verify` 명령어 또는 `verification-loop` 스킬을 커밋 전 게이트로 사용.

4. **단순 루프에서의 Ralphinho 계층 접근법** — 순차 파이프라인에서도 단순 작업은 Haiku로, 복잡한 작업은 Opus로 라우팅할 수 있습니다:
   ```bash
   # 단순 형식 수정
   claude -p --model haiku "src/utils.ts에서 임포트 순서 수정"

   # 복잡한 아키텍처 변경
   claude -p --model opus "auth 모듈을 전략 패턴 사용으로 리팩토링"
   ```

## 안티패턴

### 일반적인 실수

1. **종료 조건 없는 무한 루프** — 항상 max-runs, max-cost, max-duration, 또는 완료 신호를 가지세요.

2. **반복 간 컨텍스트 브리지 없음** — 각 `claude -p` 호출은 새로 시작합니다. `SHARED_TASK_NOTES.md` 또는 파일 시스템 상태를 사용하여 컨텍스트를 연결하세요.

3. **동일한 실패 재시도** — 반복이 실패하면 단순히 재시도하지 마세요. 오류 컨텍스트를 캡처하고 다음 시도에 전달하세요.

4. **정리 패스 대신 부정적 지시** — "X를 하지 마세요"라고 말하지 마세요. X를 제거하는 별도의 패스를 추가하세요.

5. **하나의 컨텍스트 창에 모든 에이전트** — 복잡한 워크플로우의 경우, 관심사를 다른 에이전트 프로세스로 분리하세요. 검토자는 절대 작성자가 되어서는 안 됩니다.

6. **병렬 작업에서 파일 중복 무시** — 두 병렬 에이전트가 같은 파일을 편집할 수 있다면, 병합 전략(순차 착지, 리베이스, 또는 충돌 해결)이 필요합니다.
