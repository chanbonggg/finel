# MCP 서버 평가 가이드

## 개요

이 문서는 MCP 서버에 대한 포괄적인 평가를 생성하는 방법을 안내합니다. 평가는 LLM이 제공된 툴만을 사용하여 현실적이고 복잡한 질문에 효과적으로 답할 수 있는지 테스트합니다.

---

## 빠른 참조

### 평가 요구사항
- 사람이 읽기 쉬운 질문 10개 생성
- 질문은 읽기 전용, 독립적, 비파괴적이어야 함
- 각 질문에는 여러 툴 호출 필요 (잠재적으로 수십 번)
- 답변은 단일하고 검증 가능한 값이어야 함
- 답변은 시간이 지나도 변하지 않는 안정적인 값이어야 함

### 출력 형식
```xml
<evaluation>
   <qa_pair>
      <question>질문 내용</question>
      <answer>단일 검증 가능한 답변</answer>
   </qa_pair>
</evaluation>
```

---

## 평가의 목적

MCP 서버의 품질 척도는 서버가 툴을 얼마나 잘 또는 포괄적으로 구현했는지가 아니라, 이러한 구현(입력/출력 스키마, 독스트링/설명, 기능)이 다른 컨텍스트 없이 MCP 서버에만 접근하는 LLM이 현실적이고 어려운 질문에 얼마나 잘 답할 수 있게 하는지입니다.

## 평가 개요

읽기 전용, 독립적, 비파괴적, 멱등적 작업만으로 답할 수 있는 사람이 읽기 쉬운 질문 10개를 생성합니다. 각 질문은:
- 현실적일 것
- 명확하고 간결할 것
- 모호하지 않을 것
- 복잡하여 잠재적으로 수십 번의 툴 호출 또는 단계가 필요할 것
- 사전에 파악한 단일하고 검증 가능한 값으로 답할 수 있을 것

## 질문 가이드라인

### 핵심 요구사항

1. **질문은 반드시 독립적이어야 합니다**
   - 각 질문은 다른 질문의 답변에 의존하면 안 됩니다
   - 다른 질문 처리에서 나온 이전 쓰기 작업을 전제하면 안 됩니다

2. **질문은 반드시 비파괴적이고 멱등적인 툴 사용만을 요구해야 합니다**
   - 올바른 답변을 얻기 위해 상태를 수정할 것을 지시하거나 요구하면 안 됩니다

3. **질문은 현실적이고, 명확하고, 간결하며, 복잡해야 합니다**
   - 다른 LLM이 답하기 위해 여러 (잠재적으로 수십 개의) 툴이나 단계를 사용해야 합니다

### 복잡성 및 깊이

4. **질문은 깊은 탐색을 요구해야 합니다**
   - 여러 하위 질문과 순차적 툴 호출이 필요한 다중 홉 질문 고려
   - 각 단계는 이전 질문에서 얻은 정보를 활용해야 합니다

5. **질문은 광범위한 페이징이 필요할 수 있습니다**
   - 여러 페이지의 결과를 통해 페이징이 필요할 수 있음
   - 틈새 정보를 찾기 위해 오래된 데이터(1-2년 지난) 조회가 필요할 수 있음
   - 질문은 반드시 어려워야 합니다

6. **질문은 깊은 이해를 요구해야 합니다**
   - 표면적인 지식이 아닌 깊은 이해 필요
   - 증거가 필요한 참/거짓 질문으로 복잡한 개념 제시 가능
   - LLM이 다양한 가설을 탐색해야 하는 객관식 형식 사용 가능

7. **질문은 단순한 키워드 검색으로 해결할 수 없어야 합니다**
   - 대상 콘텐츠의 특정 키워드 포함 금지
   - 동의어, 관련 개념 또는 바꿔 말하기 사용
   - 여러 번 검색하고, 여러 관련 항목을 분석하고, 컨텍스트를 추출하여 답변을 도출해야 함

### 툴 테스트

8. **질문은 툴 반환 값을 스트레스 테스트해야 합니다**
   - LLM을 압도하는 큰 JSON 객체나 목록을 반환하는 툴 유발 가능
   - 다양한 데이터 형식에 대한 이해를 요구해야 합니다:
     - ID와 이름
     - 타임스탬프와 날짜/시간 (월, 일, 년, 초)
     - 파일 ID, 이름, 확장자, MIME 타입
     - URL, GID 등
   - 모든 유용한 데이터 형식을 반환하는 툴의 능력을 탐색해야 함

9. **질문은 주로 실제 인간 사용 사례를 반영해야 합니다**
   - LLM의 도움을 받는 인간이 신경 쓸 만한 정보 검색 작업

10. **질문은 수십 번의 툴 호출을 요구할 수 있습니다**
    - 제한된 컨텍스트를 가진 LLM에게 도전
    - MCP 서버 툴이 반환하는 정보를 줄이도록 장려

11. **모호한 질문 포함**
    - 모호하거나 어떤 툴을 호출할지 어려운 결정을 요구할 수 있음
    - LLM이 실수를 하거나 잘못 해석할 가능성을 강제
    - 모호성에도 불구하고 단일하고 검증 가능한 답변이 있어야 함

### 안정성

12. **질문은 답변이 변하지 않도록 설계되어야 합니다**
    - 동적인 "현재 상태"에 의존하는 질문 금지
    - 예를 들어, 다음을 세는 질문 금지:
      - 게시물의 반응 수
      - 스레드의 답글 수
      - 채널의 멤버 수

13. **MCP 서버가 생성하는 질문의 종류를 제한하게 하지 마세요**
    - 도전적이고 복잡한 질문 생성
    - 일부는 가용한 MCP 서버 툴로 해결 불가능할 수 있음
    - 특정 출력 형식 요구 가능 (날짜/시간 대 에폭 시간, JSON 대 MARKDOWN)
    - 완료하는 데 수십 번의 툴 호출이 필요한 질문 가능

## 답변 가이드라인

### 검증

1. **답변은 직접 문자열 비교로 검증 가능해야 합니다**
   - 답변이 여러 형식으로 다시 쓰일 수 있다면 질문에 출력 형식을 명확히 명시
   - 예: "YYYY/MM/DD를 사용하세요.", "True 또는 False로 답하세요.", "A, B, C, D 중 하나만 답하세요."
   - 답변은 단일 검증 가능한 값이어야 합니다:
     - 사용자 ID, 사용자 이름, 표시 이름, 이름, 성
     - 채널 ID, 채널 이름
     - 메시지 ID, 문자열
     - URL, 제목
     - 숫자 수량
     - 타임스탬프, 날짜/시간
     - 불리언 (참/거짓 질문용)
     - 이메일 주소, 전화번호
     - 파일 ID, 파일 이름, 파일 확장자
     - 객관식 답변
   - 답변에 특별한 포맷팅이나 복잡하고 구조화된 출력이 불필요해야 함
   - 답변은 직접 문자열 비교로 검증됨

### 가독성

2. **답변은 일반적으로 사람이 읽기 쉬운 형식을 선호해야 합니다**
   - 예: 이름, 이름, 성, 날짜/시간, 파일 이름, 메시지 문자열, URL, 예/아니오, 참/거짓, a/b/c/d
   - 불투명한 ID보다 사람이 읽을 수 있는 형식 선호 (ID도 허용)
   - 대부분의 답변은 사람이 읽을 수 있어야 합니다

### 안정성

3. **답변은 안정적/고정적이어야 합니다**
   - 오래된 콘텐츠 참조 (예: 종료된 대화, 출시된 프로젝트, 답변된 질문)
   - 항상 동일한 답변을 반환하는 "종료된" 개념을 기반으로 질문 생성
   - 일정한 시간 창을 고려하도록 질문할 수 있음
   - 변경될 가능성이 낮은 컨텍스트에 의존
   - 예: 논문 이름을 찾는 경우, 나중에 발표된 논문과 혼동되지 않을 만큼 충분히 구체적이어야 함

4. **답변은 명확하고 모호하지 않아야 합니다**
   - 질문은 단일하고 명확한 답변이 있도록 설계되어야 함
   - MCP 서버 툴을 사용하여 답변을 도출할 수 있어야 함

### 다양성

5. **답변은 다양해야 합니다**
   - 답변은 다양한 형식과 형태의 단일 검증 가능한 값이어야 합니다
   - 사용자 개념: 사용자 ID, 사용자 이름, 표시 이름, 이름, 성, 이메일 주소, 전화번호
   - 채널 개념: 채널 ID, 채널 이름, 채널 주제
   - 메시지 개념: 메시지 ID, 메시지 문자열, 타임스탬프, 월, 일, 년

6. **답변은 복잡한 구조가 아니어야 합니다**
   - 값의 목록이 아닐 것
   - 복잡한 객체가 아닐 것
   - ID나 문자열의 목록이 아닐 것
   - 자연어 텍스트가 아닐 것
   - 직접 문자열 비교로 간단히 검증 가능한 경우는 예외
   - 현실적으로 재현 가능해야 함
   - LLM이 동일한 목록을 다른 순서나 형식으로 반환할 가능성이 낮아야 함

## 평가 프로세스

### 1단계: 문서 검사

대상 API의 문서를 읽어 다음을 이해합니다:
- 사용 가능한 엔드포인트와 기능
- 모호함이 있다면 웹에서 추가 정보 수집
- 가능한 한 많이 병렬화
- 각 서브에이전트는 파일 시스템이나 웹의 문서만 검사해야 함

### 2단계: 툴 검사

MCP 서버에서 사용 가능한 툴 나열:
- MCP 서버를 직접 검사
- 입력/출력 스키마, 독스트링, 설명 이해
- 이 단계에서는 툴 자체를 호출하지 않음

### 3단계: 이해 발전

충분한 이해를 얻을 때까지 1단계와 2단계를 반복합니다:
- 여러 번 반복
- 생성하고 싶은 작업 종류에 대해 생각
- 이해를 다듬기
- MCP 서버 구현 코드 자체는 절대 읽지 않음
- 직관과 이해를 사용하여 합리적이고 현실적이지만 매우 도전적인 작업 생성

### 4단계: 읽기 전용 콘텐츠 검사

API와 툴을 이해한 후 MCP 서버 툴을 사용합니다:
- 읽기 전용 및 비파괴적 작업만으로 콘텐츠 검사
- 목표: 현실적인 질문 생성을 위한 특정 콘텐츠(사용자, 채널, 메시지, 프로젝트, 작업) 식별
- 상태를 수정하는 툴은 호출하지 않음
- MCP 서버 구현 코드 자체는 읽지 않음
- 독립적인 탐색을 추구하는 개별 서브에이전트로 이 단계를 병렬화
- 각 서브에이전트가 읽기 전용, 비파괴적, 멱등적 작업만 수행하도록 주의
- 주의: 일부 툴은 컨텍스트를 소진시킬 수 있는 대량의 데이터를 반환할 수 있음
- 탐색을 위해 점진적이고 작고 목표화된 툴 호출 수행
- 모든 툴 호출 요청에서 `limit` 파라미터를 사용하여 결과 제한 (<10)
- 페이지네이션 사용

### 5단계: 작업 생성

콘텐츠를 검사한 후 사람이 읽기 쉬운 질문 10개를 생성합니다:
- LLM이 MCP 서버로 답할 수 있어야 함
- 위의 모든 질문 및 답변 가이드라인 준수

## 출력 형식

각 QA 쌍은 질문과 답변으로 구성됩니다. 출력은 다음 구조의 XML 파일이어야 합니다:

```xml
<evaluation>
   <qa_pair>
      <question>Q2 2024에 생성된 완료된 작업 수가 가장 많은 프로젝트를 찾으세요. 프로젝트 이름이 무엇인가요?</question>
      <answer>Website Redesign</answer>
   </qa_pair>
   <qa_pair>
      <question>"bug"로 라벨이 지정되어 2024년 3월에 종료된 이슈를 검색하세요. 가장 많은 이슈를 종료한 사용자는 누구인가요? 사용자 이름을 제공하세요.</question>
      <answer>sarah_dev</answer>
   </qa_pair>
   <qa_pair>
      <question>/api 디렉토리의 파일을 수정하고 2024년 1월 1일부터 31일 사이에 병합된 풀 리퀘스트를 찾으세요. 이 PR에 기여한 서로 다른 기여자는 몇 명인가요?</question>
      <answer>7</answer>
   </qa_pair>
   <qa_pair>
      <question>2023년 이전에 생성된 별이 가장 많은 저장소를 찾으세요. 저장소 이름이 무엇인가요?</question>
      <answer>data-pipeline</answer>
   </qa_pair>
</evaluation>
```

## 평가 예시

### 좋은 질문

**예시 1: 깊은 탐색이 필요한 다중 홉 질문 (GitHub MCP)**
```xml
<qa_pair>
   <question>2023년 3분기에 아카이브된 저장소 중 이전에 조직에서 가장 많이 포크된 프로젝트였던 것을 찾으세요. 그 저장소에서 사용된 주요 프로그래밍 언어는 무엇이었나요?</question>
   <answer>Python</answer>
</qa_pair>
```

이 질문이 좋은 이유:
- 아카이브된 저장소를 찾기 위한 여러 번의 검색 필요
- 아카이브 전 가장 많이 포크된 저장소 식별 필요
- 저장소 세부 정보에서 언어 확인 필요
- 답변이 단순하고 검증 가능한 값
- 변경되지 않는 역사적(종료된) 데이터 기반

**예시 2: 키워드 매칭 없이 컨텍스트 이해가 필요한 질문 (프로젝트 관리 MCP)**
```xml
<qa_pair>
   <question>2023년 말에 완료된 고객 온보딩 개선에 초점을 맞춘 이니셔티브를 찾으세요. 프로젝트 리더는 완료 후 회고 문서를 작성했습니다. 당시 리더의 직책이 무엇이었나요?</question>
   <answer>Product Manager</answer>
</qa_pair>
```

이 질문이 좋은 이유:
- 특정 프로젝트 이름이 아닌 설명("고객 온보딩 개선에 초점") 사용
- 특정 기간의 완료된 프로젝트 찾기 필요
- 프로젝트 리더와 그 직책 식별 필요
- 회고 문서에서 컨텍스트 이해 필요
- 답변이 사람이 읽기 쉽고 안정적
- 완료된 작업 기반 (변경되지 않음)

**예시 3: 여러 단계가 필요한 복잡한 집계 (이슈 트래커 MCP)**
```xml
<qa_pair>
   <question>2024년 1월에 보고된 모든 버그 중 중요 우선순위로 표시된 것들 중에서, 담당자 중 48시간 이내에 할당된 버그를 가장 높은 비율로 해결한 사람은 누구인가요? 담당자의 사용자 이름을 제공하세요.</question>
   <answer>alex_eng</answer>
</qa_pair>
```

이 질문이 좋은 이유:
- 날짜, 우선순위, 상태로 버그 필터링 필요
- 담당자별로 그룹화하고 해결 비율 계산 필요
- 48시간 창을 결정하기 위한 타임스탬프 이해 필요
- 페이지네이션 테스트 (잠재적으로 많은 버그 처리)
- 단일 사용자 이름이 답변
- 특정 기간의 역사적 데이터 기반

**예시 4: 여러 데이터 타입에 걸친 종합이 필요한 질문 (CRM MCP)**
```xml
<qa_pair>
   <question>2023년 4분기에 Starter에서 Enterprise 플랜으로 업그레이드하고 연간 계약 가치가 가장 높은 계정을 찾으세요. 이 계정은 어떤 업종에서 운영되나요?</question>
   <answer>Healthcare</answer>
</qa_pair>
```

이 질문이 좋은 이유:
- 구독 등급 변경 이해 필요
- 특정 기간의 업그레이드 이벤트 식별 필요
- 계약 가치 비교 필요
- 계정 업종 정보 접근 필요
- 답변이 단순하고 검증 가능
- 완료된 역사적 거래 기반

### 나쁜 질문

**예시 1: 시간에 따라 답변이 변함**
```xml
<qa_pair>
   <question>현재 엔지니어링 팀에 할당된 열린 이슈는 몇 개인가요?</question>
   <answer>47</answer>
</qa_pair>
```

이 질문이 나쁜 이유:
- 이슈가 생성, 종료, 재할당됨에 따라 답변이 변함
- 안정적/고정적인 데이터 기반이 아님
- 동적인 "현재 상태"에 의존

**예시 2: 키워드 검색으로 너무 쉽게 해결 가능**
```xml
<qa_pair>
   <question>"Add authentication feature"라는 제목의 풀 리퀘스트를 찾아 누가 생성했는지 알려주세요.</question>
   <answer>developer123</answer>
</qa_pair>
```

이 질문이 나쁜 이유:
- 정확한 제목의 단순한 키워드 검색으로 해결 가능
- 깊은 탐색이나 이해 불필요
- 종합이나 분석 불필요

**예시 3: 모호한 답변 형식**
```xml
<qa_pair>
   <question>주요 언어가 Python인 모든 저장소를 나열하세요.</question>
   <answer>repo1, repo2, repo3, data-pipeline, ml-tools</answer>
</qa_pair>
```

이 질문이 나쁜 이유:
- 답변이 임의의 순서로 반환될 수 있는 목록
- 직접 문자열 비교로 검증하기 어려움
- LLM이 다르게 포맷할 수 있음 (JSON 배열, 쉼표 구분, 줄바꿈 구분)
- 특정 집계(개수) 또는 최상위(별 가장 많음)를 묻는 것이 더 나음

## 검증 프로세스

평가를 생성한 후:

1. **XML 파일을 검사**하여 스키마를 이해
2. **각 작업 지침을 로드**하고 MCP 서버와 툴을 사용하여 직접 작업을 해결하면서 올바른 답변을 병렬로 식별
3. **쓰기 또는 파괴적 작업이 필요한 작업 표시**
4. **모든 올바른 답변을 수집**하고 문서의 잘못된 답변 교체
5. **쓰기 또는 파괴적 작업이 필요한 `<qa_pair>` 제거**

컨텍스트를 소진하지 않도록 작업을 병렬로 해결하고, 모든 답변을 수집한 후 파일을 마지막에 변경합니다.

## 고품질 평가 생성 팁

1. **시작 전에 충분히 생각하고 계획 세우기**
2. **기회가 있을 때마다 병렬화**하여 프로세스 속도를 높이고 컨텍스트 관리
3. **현실적인 사용 사례에 집중** — 인간이 실제로 달성하고 싶은 것
4. **도전적인 질문 생성** — MCP 서버 기능의 한계를 테스트
5. **역사적 데이터와 종료된 개념**으로 안정성 확보
6. **MCP 서버 툴로 질문을 직접 해결하여 답변 검증**
7. **프로세스 중 학습한 내용을 기반으로 반복 및 개선**

# 평가 실행

평가 파일을 생성한 후 제공된 평가 하네스를 사용하여 MCP 서버를 테스트할 수 있습니다.

## 설정

1. **의존성 설치**

   ```bash
   pip install -r scripts/requirements.txt
   ```

   또는 수동으로 설치:
   ```bash
   pip install anthropic mcp
   ```

2. **API 키 설정**

   ```bash
   export ANTHROPIC_API_KEY=your_api_key_here
   ```

## 평가 파일 형식

평가 파일은 `<qa_pair>` 요소를 가진 XML 형식을 사용합니다:

```xml
<evaluation>
   <qa_pair>
      <question>Q2 2024에 생성된 완료된 작업 수가 가장 많은 프로젝트를 찾으세요. 프로젝트 이름이 무엇인가요?</question>
      <answer>Website Redesign</answer>
   </qa_pair>
   <qa_pair>
      <question>"bug"로 라벨이 지정되어 2024년 3월에 종료된 이슈를 검색하세요. 가장 많은 이슈를 종료한 사용자는 누구인가요? 사용자 이름을 제공하세요.</question>
      <answer>sarah_dev</answer>
   </qa_pair>
</evaluation>
```

## 평가 실행

평가 스크립트(`scripts/evaluation.py`)는 세 가지 트랜스포트 타입을 지원합니다:

**중요:**
- **stdio 트랜스포트**: 평가 스크립트가 자동으로 MCP 서버 프로세스를 실행합니다. 서버를 수동으로 실행하지 마세요.
- **sse/http 트랜스포트**: 평가 실행 전에 MCP 서버를 별도로 시작해야 합니다. 스크립트는 지정된 URL에서 이미 실행 중인 서버에 연결합니다.

### 1. 로컬 STDIO 서버

로컬에서 실행되는 MCP 서버용 (스크립트가 자동으로 서버 실행):

```bash
python scripts/evaluation.py \
  -t stdio \
  -c python \
  -a my_mcp_server.py \
  evaluation.xml
```

환경 변수와 함께:
```bash
python scripts/evaluation.py \
  -t stdio \
  -c python \
  -a my_mcp_server.py \
  -e API_KEY=abc123 \
  -e DEBUG=true \
  evaluation.xml
```

### 2. Server-Sent Events (SSE)

SSE 기반 MCP 서버용 (먼저 서버를 시작해야 함):

```bash
python scripts/evaluation.py \
  -t sse \
  -u https://example.com/mcp \
  -H "Authorization: Bearer token123" \
  -H "X-Custom-Header: value" \
  evaluation.xml
```

### 3. HTTP (Streamable HTTP)

HTTP 기반 MCP 서버용 (먼저 서버를 시작해야 함):

```bash
python scripts/evaluation.py \
  -t http \
  -u https://example.com/mcp \
  -H "Authorization: Bearer token123" \
  evaluation.xml
```

## 커맨드라인 옵션

```
사용법: evaluation.py [-h] [-t {stdio,sse,http}] [-m MODEL] [-c COMMAND]
                     [-a ARGS [ARGS ...]] [-e ENV [ENV ...]] [-u URL]
                     [-H HEADERS [HEADERS ...]] [-o OUTPUT]
                     eval_file

위치 인수:
  eval_file             평가 XML 파일 경로

선택적 인수:
  -h, --help            도움말 메시지 표시
  -t, --transport       트랜스포트 타입: stdio, sse, http (기본값: stdio)
  -m, --model           사용할 Claude 모델 (기본값: claude-3-7-sonnet-20250219)
  -o, --output          보고서 출력 파일 (기본값: 표준 출력)

stdio 옵션:
  -c, --command         MCP 서버 실행 명령 (예: python, node)
  -a, --args            명령 인수 (예: server.py)
  -e, --env             KEY=VALUE 형식의 환경 변수

sse/http 옵션:
  -u, --url             MCP 서버 URL
  -H, --header          'Key: Value' 형식의 HTTP 헤더
```

## 출력

평가 스크립트는 다음을 포함한 상세 보고서를 생성합니다:

- **요약 통계**:
  - 정확도 (정답/전체)
  - 평균 작업 소요 시간
  - 작업당 평균 툴 호출 수
  - 전체 툴 호출 수

- **작업별 결과**:
  - 프롬프트와 예상 응답
  - 에이전트의 실제 응답
  - 답변 정확 여부 (정답/오답)
  - 소요 시간 및 툴 호출 세부 정보
  - 에이전트의 접근 방식 요약
  - 툴에 대한 에이전트의 피드백

## 문제 해결

### 연결 오류

연결 오류가 발생하면:
- **STDIO**: 명령과 인수가 올바른지 확인
- **SSE/HTTP**: URL이 접근 가능하고 헤더가 올바른지 확인
- 필요한 API 키가 환경 변수나 헤더에 설정되어 있는지 확인

### 낮은 정확도

많은 평가가 실패하는 경우:
- 각 작업에 대한 에이전트의 피드백 검토
- 툴 설명이 명확하고 포괄적인지 확인
- 입력 파라미터가 잘 문서화되어 있는지 확인
- 툴이 너무 많거나 너무 적은 데이터를 반환하는지 확인
- 에러 메시지가 실행 가능한지 확인

### 타임아웃 문제

작업이 타임아웃되는 경우:
- 더 유능한 모델 사용 (예: `claude-3-7-sonnet-20250219`)
- 툴이 너무 많은 데이터를 반환하는지 확인
- 페이지네이션이 올바르게 작동하는지 확인
- 복잡한 질문 단순화 고려
