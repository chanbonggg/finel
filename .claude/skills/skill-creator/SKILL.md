# Skill Creator

새로운 스킬을 만들고 반복적으로 개선하기 위한 스킬입니다.

높은 수준에서 스킬 생성 프로세스는 다음과 같습니다:

- 스킬이 무엇을 해야 하는지, 그리고 대략 어떻게 해야 하는지 결정
- 스킬 초안 작성
- 몇 가지 테스트 프롬프트를 만들어 스킬에 접근권이 있는 Claude로 실행
- 사용자가 정성적/정량적으로 결과를 평가하도록 도움
  - 실행이 백그라운드에서 진행되는 동안, 정량적 평가가 없으면 초안 작성 (있으면 기존 것을 그대로 사용하거나 변경이 필요하면 수정). 그런 다음 사용자에게 설명 (이미 존재하는 경우, 기존 것을 설명)
  - `eval-viewer/generate_review.py` 스크립트를 사용하여 사용자에게 결과를 보여주고, 정량적 지표도 확인할 수 있게 함
- 사용자의 평가 피드백을 기반으로 스킬 재작성 (정량적 벤치마크에서 명백한 결함이 있는 경우도 포함)
- 만족할 때까지 반복
- 테스트 세트를 확장하여 더 큰 규모로 다시 시도

이 스킬을 사용할 때의 역할은 사용자가 이 프로세스의 어느 단계에 있는지 파악하고, 이 단계들을 진행하도록 도움을 주는 것입니다. 예를 들어, 사용자가 "X를 위한 스킬을 만들고 싶어"라고 하면, 의미를 좁혀나가고, 초안을 작성하고, 테스트 케이스를 작성하고, 평가 방법을 결정하고, 모든 프롬프트를 실행하고, 반복할 수 있습니다.

반면에 사용자가 이미 스킬 초안을 가지고 있을 수도 있습니다. 이 경우 평가/반복 부분으로 바로 넘어갈 수 있습니다.

물론, 항상 유연하게 대응해야 하며 사용자가 "많은 평가를 실행할 필요 없어, 그냥 같이 해봐"라고 하면 그렇게 할 수도 있습니다.

그런 다음 스킬이 완성된 후 (하지만 순서는 유연함), 스킬 description 개선기를 실행할 수 있습니다. 이를 위한 별도의 스크립트가 있으며, 스킬 트리거를 최적화합니다.

## 사용자와의 소통

Skill creator는 코딩 용어에 대한 친숙도가 다양한 사람들이 사용할 수 있습니다. 최근에 Claude의 힘에 영감을 받은 배관공이 터미널을 열고, 부모님과 조부모님이 "npm 설치 방법"을 구글링하기 시작하는 추세가 있습니다. 반면에 대부분의 사용자는 상당히 컴퓨터에 익숙할 것입니다.

따라서 컨텍스트 단서에 주의를 기울여 소통 방식을 결정하세요! 기본적으로 예를 들면:

- "evaluation"과 "benchmark"는 경계선이지만 사용 가능
- "JSON"과 "assertion"은 사용자가 그것들을 알고 있다는 진지한 단서를 보기 전까지 설명 없이 사용하지 마세요

확실하지 않으면 간단히 용어를 설명해도 됩니다. 사용자가 이해할지 불확실하면 짧은 정의와 함께 용어를 명확히 해도 됩니다.

## 스킬 생성

### 의도 파악

사용자의 의도를 이해하는 것부터 시작합니다. 현재 대화에 이미 사용자가 포착하려는 워크플로가 있을 수 있습니다 (예: "이것을 스킬로 만들어줘"). 있다면 대화 히스토리에서 먼저 답을 추출합니다 — 사용된 도구, 단계 순서, 사용자가 수정한 내용, 관찰된 입력/출력 형식. 사용자가 빈칸을 채워야 할 수도 있으며, 다음 단계로 넘어가기 전에 확인을 받아야 합니다.

1. 이 스킬로 Claude가 무엇을 할 수 있어야 하나요?
2. 이 스킬은 언제 트리거되어야 하나요? (어떤 사용자 문구/컨텍스트)
3. 예상되는 출력 형식은 무엇인가요?
4. 스킬이 작동하는지 확인하기 위한 테스트 케이스를 설정해야 할까요? 객관적으로 검증 가능한 출력이 있는 스킬 (파일 변환, 데이터 추출, 코드 생성, 고정된 워크플로 단계)은 테스트 케이스에서 이점을 얻습니다. 주관적인 출력이 있는 스킬 (작성 스타일, 예술)은 종종 테스트 케이스가 필요하지 않습니다. 스킬 유형에 따라 적절한 기본값을 제안하되, 사용자가 결정하게 합니다.

### 인터뷰 및 리서치

엣지 케이스, 입력/출력 형식, 예시 파일, 성공 기준, 의존성에 대해 적극적으로 질문합니다. 이 부분이 정리될 때까지 테스트 프롬프트 작성을 기다립니다.

사용 가능한 MCP를 확인합니다 - 리서치에 유용하면 (문서 검색, 유사 스킬 찾기, 모범 사례 조회), 서브에이전트를 통해 병렬로 리서치하거나 (사용 가능한 경우) 인라인으로 합니다. 사용자의 부담을 줄이기 위해 컨텍스트를 준비해 옵니다.

### SKILL.md 작성

사용자 인터뷰를 바탕으로 다음 구성 요소를 채웁니다:

- **name**: 스킬 식별자
- **description**: 트리거 시점과 수행 내용. 이것이 주요 트리거 메커니즘입니다 - 스킬이 무엇을 하는지와 언제 사용하는지 특정 컨텍스트를 모두 포함합니다. 모든 "사용 시점" 정보는 여기에 작성하고, 본문에는 넣지 않습니다. 참고: 현재 Claude는 스킬을 "과소트리거"하는 경향이 있습니다 — 유용할 때도 사용하지 않는 경향. 이를 해결하기 위해 스킬 description을 약간 "적극적으로" 만드세요. 예를 들어, "내부 Anthropic 데이터를 표시하는 간단하고 빠른 대시보드 구축 방법" 대신 "내부 Anthropic 데이터를 표시하는 간단하고 빠른 대시보드 구축 방법. 사용자가 대시보드, 데이터 시각화, 내부 지표를 언급하거나 회사 데이터를 표시하고 싶을 때 명시적으로 '대시보드'를 요청하지 않더라도 이 스킬을 반드시 사용합니다."
- **compatibility**: 필요한 도구, 의존성 (선택적, 드물게 필요)
- **나머지 스킬 :)**

### 스킬 작성 가이드

#### 스킬의 구조

```
skill-name/
├── SKILL.md (필수)
│   ├── YAML frontmatter (name, description 필수)
│   └── Markdown 지시사항
└── 번들된 리소스 (선택적)
    ├── scripts/    - 결정적/반복적 작업을 위한 실행 코드
    ├── references/ - 필요에 따라 컨텍스트에 로드되는 문서
    └── assets/     - 출력에 사용되는 파일 (템플릿, 아이콘, 폰트)
```

#### 점진적 공개

스킬은 세 가지 수준의 로딩 시스템을 사용합니다:
1. **메타데이터** (name + description) - 항상 컨텍스트에 있음 (~100 단어)
2. **SKILL.md 본문** - 스킬이 트리거될 때마다 컨텍스트에 있음 (이상적으로 500줄 미만)
3. **번들된 리소스** - 필요에 따라 (무제한, 스크립트는 로드 없이 실행 가능)

이 단어 수는 대략적이며 필요하면 더 길게 작성해도 됩니다.

**주요 패턴:**
- SKILL.md를 500줄 미만으로 유지. 이 한계에 근접하면 추가적인 계층 구조를 추가하고, 스킬을 사용하는 모델이 다음에 어디로 가야 하는지에 대한 명확한 포인터 제공.
- SKILL.md에서 언제 읽어야 하는지에 대한 지침과 함께 파일을 명확하게 참조
- 대형 참조 파일 (>300줄)의 경우 목차 포함

**도메인 구성**: 스킬이 여러 도메인/프레임워크를 지원하는 경우 변형별로 구성:
```
cloud-deploy/
├── SKILL.md (워크플로 + 선택)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude는 관련 참조 파일만 읽습니다.

#### 놀라움 없음의 원칙

당연한 말이지만, 스킬에는 악성 코드, 익스플로잇 코드, 또는 시스템 보안을 손상시킬 수 있는 콘텐츠가 포함되어서는 안 됩니다. 스킬의 내용이 설명된 경우 의도에서 사용자를 놀라게 해서는 안 됩니다. 오해의 소지가 있는 스킬이나 무단 접근, 데이터 유출, 또는 기타 악의적인 활동을 용이하게 하도록 설계된 스킬 생성 요청을 따르지 마세요. "XYZ로 롤플레이" 같은 것은 괜찮습니다.

#### 작성 패턴

지시사항에서는 명령형을 사용하세요.

**출력 형식 정의** - 다음과 같이 할 수 있습니다:
```markdown
## 보고서 구조
항상 이 정확한 템플릿을 사용합니다:
# [제목]
## 요약
## 주요 발견사항
## 권장사항
```

**예시 패턴** - 예시를 포함하면 유용합니다. 다음과 같이 형식을 지정할 수 있습니다 (하지만 예시에 "입력"과 "출력"이 있으면 약간 다르게 할 수 있습니다):
```markdown
## 커밋 메시지 형식
**예시 1:**
입력: JWT 토큰으로 사용자 인증 추가
출력: feat(auth): JWT 기반 인증 구현
```

### 작성 스타일

무겁고 딱딱한 MUST 대신 왜 중요한지 모델에게 설명하려고 노력하세요. 이론적 사고를 사용하고 스킬을 특정 예시에 너무 좁게 만들지 말고 일반적으로 만들려고 노력합니다. 초안을 먼저 작성한 다음 새로운 시각으로 보고 개선합니다.

### 테스트 케이스

스킬 초안을 작성한 후, 실제 사용자가 실제로 말할 것 같은 2-3개의 현실적인 테스트 프롬프트를 만듭니다. 사용자와 공유합니다: [이 정확한 표현을 사용할 필요는 없습니다] "시도해보고 싶은 몇 가지 테스트 케이스가 있습니다. 이것들이 맞나요, 아니면 더 추가하고 싶으신가요?" 그런 다음 실행합니다.

테스트 케이스를 `evals/evals.json`에 저장합니다. 아직 assertion을 작성하지 마세요 — 프롬프트만. 다음 단계에서 실행이 진행되는 동안 assertion을 초안 작성합니다.

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "사용자의 작업 프롬프트",
      "expected_output": "예상 결과 설명",
      "files": []
    }
  ]
}
```

전체 스키마 (`assertions` 필드 포함)는 `references/schemas.md`를 참조하세요.

## 테스트 케이스 실행 및 평가

이 섹션은 하나의 연속적인 시퀀스입니다 — 중간에 멈추지 마세요. `/skill-test` 또는 다른 테스트 스킬을 사용하지 마세요.

결과를 스킬 디렉토리의 형제 디렉토리인 `<skill-name>-workspace/`에 넣습니다. 워크스페이스 내에서 반복 별로 결과를 구성하고 (`iteration-1/`, `iteration-2/` 등), 그 안에서 각 테스트 케이스는 디렉토리를 갖습니다 (`eval-0/`, `eval-1/` 등). 미리 모든 것을 만들지 마세요 — 진행하면서 디렉토리를 만드세요.

### 1단계: 같은 턴에서 모든 실행 스폰 (with-skill과 baseline 모두)

각 테스트 케이스에 대해 같은 턴에서 두 개의 서브에이전트를 스폰합니다 — 하나는 스킬로, 하나는 없이. 중요: with-skill 실행을 먼저 스폰하고 나중에 baseline으로 돌아오지 마세요. 모든 것이 거의 동시에 완료될 수 있도록 한꺼번에 실행합니다.

**with-skill 실행:**

```
이 작업을 실행합니다:
- 스킬 경로: <path-to-skill>
- 작업: <eval prompt>
- 입력 파일: <eval files if any, or "없음">
- 출력 저장 위치: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 저장할 출력: <사용자가 관심 있는 것 — 예: ".docx 파일", "최종 CSV">
```

**Baseline 실행** (동일한 프롬프트, 하지만 baseline은 컨텍스트에 따라 다름):
- **새 스킬 생성 시**: 스킬 없음. 동일한 프롬프트, 스킬 경로 없음, `without_skill/outputs/`에 저장.
- **기존 스킬 개선 시**: 이전 버전. 편집 전에 스킬을 스냅샷 (`cp -r <skill-path> <workspace>/skill-snapshot/`)한 다음 baseline 서브에이전트를 스냅샷으로 지정. `old_skill/outputs/`에 저장.

각 테스트 케이스에 대해 `eval_metadata.json`을 작성합니다 (assertion은 지금 비워도 됨). 테스트하는 내용을 기반으로 각 평가에 설명적인 이름을 부여합니다 — "eval-0"이 아닌. 이 이름을 디렉토리에도 사용합니다. 이 반복이 새롭거나 수정된 eval 프롬프트를 사용하는 경우, 각 새 eval 디렉토리에 대해 이 파일을 만듭니다 — 이전 반복에서 넘어온다고 가정하지 마세요.

```json
{
  "eval_id": 0,
  "eval_name": "여기에-설명적인-이름",
  "prompt": "사용자의 작업 프롬프트",
  "assertions": []
}
```

### 2단계: 실행 진행 중 assertion 초안 작성

실행이 완료될 때까지 기다리지 마세요 — 이 시간을 생산적으로 사용할 수 있습니다. 각 테스트 케이스에 대한 정량적 assertion을 초안 작성하고 사용자에게 설명합니다. `evals/evals.json`에 이미 assertion이 있으면 검토하고 무엇을 확인하는지 설명합니다.

좋은 assertion은 객관적으로 검증 가능하고 설명적인 이름을 가집니다 — 벤치마크 뷰어에서 명확하게 읽혀야 결과를 한눈에 보는 사람이 각 assertion이 무엇을 확인하는지 즉시 이해할 수 있습니다. 주관적인 스킬 (작성 스타일, 디자인 품질)은 정성적으로 평가하는 것이 더 좋습니다 — 인간의 판단이 필요한 것에 assertion을 억지로 적용하지 마세요.

초안이 완성되면 `eval_metadata.json` 파일과 `evals/evals.json`을 assertion으로 업데이트합니다. 뷰어에서 무엇을 볼 수 있는지도 사용자에게 설명합니다 — 정성적 출력과 정량적 벤치마크 모두.

### 3단계: 실행이 완료되면 타이밍 데이터 캡처

각 서브에이전트 작업이 완료되면 `total_tokens`와 `duration_ms`가 포함된 알림을 받습니다. 즉시 이 데이터를 실행 디렉토리의 `timing.json`에 저장합니다:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

이것은 이 데이터를 캡처할 유일한 기회입니다 — 작업 알림을 통해 전달되며 다른 곳에는 저장되지 않습니다. 배치로 처리하려고 하지 말고 각 알림이 도착하는 즉시 처리합니다.

### 4단계: 채점, 집계, 뷰어 실행

모든 실행이 완료되면:

1. **각 실행 채점** — `agents/grader.md`를 읽고 각 assertion을 출력에 대해 평가하는 채점 서브에이전트를 스폰합니다 (또는 인라인으로 채점). 각 실행 디렉토리의 `grading.json`에 결과를 저장합니다. grading.json의 expectations 배열은 `text`, `passed`, `evidence` 필드를 사용해야 합니다 (`name`/`met`/`details` 또는 다른 변형이 아님) — 뷰어는 이 정확한 필드 이름에 의존합니다. 프로그래밍 방식으로 확인할 수 있는 assertion의 경우 눈으로 보지 말고 스크립트를 작성하고 실행합니다 — 스크립트는 더 빠르고 더 신뢰할 수 있으며 여러 반복에 걸쳐 재사용할 수 있습니다.

2. **벤치마크로 집계** — skill-creator 디렉토리에서 집계 스크립트를 실행합니다:
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   이는 각 구성에 대해 pass_rate, time, tokens와 평균 ± 표준편차 및 델타가 있는 `benchmark.json`과 `benchmark.md`를 생성합니다. benchmark.json을 수동으로 생성하는 경우 뷰어가 예상하는 정확한 스키마는 `references/schemas.md`를 참조합니다.
각 with_skill 버전을 baseline 상대 버전 앞에 놓습니다.

3. **분석가 패스 실행** — 벤치마크 데이터를 읽고 집계 통계가 숨길 수 있는 패턴을 표면화합니다. 무엇을 찾아야 하는지는 `agents/analyzer.md` ("벤치마크 결과 분석" 섹션)를 참조합니다 — 스킬에 관계없이 항상 통과하는 assertion (비변별적), 높은 분산 eval (불안정할 수 있음), 시간/토큰 트레이드오프 등.

4. **뷰어 실행** — 정성적 출력과 정량적 데이터 모두 포함:
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   반복 2+의 경우 `--previous-workspace <workspace>/iteration-<N-1>`도 전달합니다.

   **Cowork / headless 환경:** `webbrowser.open()`을 사용할 수 없거나 환경에 디스플레이가 없는 경우, 서버를 시작하는 대신 `--static <output_path>`를 사용하여 독립형 HTML 파일을 작성합니다. 사용자가 "모든 리뷰 제출"을 클릭하면 피드백이 `feedback.json` 파일로 다운로드됩니다. 다운로드 후 다음 반복이 선택할 수 있도록 `feedback.json`을 workspace 디렉토리에 복사합니다.

참고: 뷰어를 만들려면 generate_review.py를 사용하세요. 커스텀 HTML을 작성할 필요가 없습니다.

5. **사용자에게 알림**: "브라우저에서 결과를 열었습니다. 두 개의 탭이 있습니다 — '출력'은 각 테스트 케이스를 클릭하고 피드백을 남길 수 있게 해주며, '벤치마크'는 정량적 비교를 보여줍니다. 완료되면 돌아와서 알려주세요."

### 사용자가 뷰어에서 보는 것

"출력" 탭은 한 번에 하나의 테스트 케이스를 보여줍니다:
- **프롬프트**: 주어진 작업
- **출력**: 스킬이 생성한 파일, 가능한 경우 인라인으로 렌더링
- **이전 출력** (반복 2+): 이전 반복 출력을 보여주는 접힌 섹션
- **공식 채점** (채점이 실행된 경우): assertion 통과/실패를 보여주는 접힌 섹션
- **피드백**: 입력 시 자동 저장되는 텍스트박스
- **이전 피드백** (반복 2+): 지난번 댓글, 텍스트박스 아래에 표시

"벤치마크" 탭은 통계 요약을 보여줍니다: 각 구성에 대한 통과율, 타이밍, 토큰 사용량, 평가별 분류 및 분석가 관찰.

탐색은 이전/다음 버튼 또는 화살표 키를 통해 이루어집니다. 완료되면 "모든 리뷰 제출"을 클릭하면 모든 피드백이 `feedback.json`에 저장됩니다.

### 5단계: 피드백 읽기

사용자가 완료했다고 알려주면 `feedback.json`을 읽습니다:

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "차트에 축 레이블이 없음", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "완벽해, 마음에 들어", "timestamp": "..."}
  ],
  "status": "complete"
}
```

빈 피드백은 사용자가 괜찮다고 생각했다는 의미입니다. 사용자가 구체적인 불만을 제기한 테스트 케이스에 집중합니다.

완료되면 뷰어 서버를 종료합니다:

```bash
kill $VIEWER_PID 2>/dev/null
```

## 스킬 개선

이것이 루프의 핵심입니다. 테스트 케이스를 실행하고, 사용자가 결과를 검토했으며, 이제 피드백을 바탕으로 스킬을 더 좋게 만들어야 합니다.

### 개선에 대한 사고 방식

1. **피드백에서 일반화합니다.** 여기서 일어나는 큰 그림은 백만 번 (아마 말 그대로, 아마 더 많이) 여러 다른 프롬프트에 걸쳐 사용될 수 있는 스킬을 만들려고 한다는 것입니다. 여기서 사용자와 함께 몇 가지 예시만 반복하는 이유는 더 빨리 이동하는 데 도움이 되기 때문입니다. 사용자는 이 예시들을 속속들이 알고 있으며 새로운 출력을 빠르게 평가할 수 있습니다. 하지만 사용자와 공동 개발하는 스킬이 그 예시들에만 작동한다면 쓸모가 없습니다. 까다롭고 과적합된 변경을 하거나 억압적으로 제한적인 MUST를 넣는 대신, 고집스러운 문제가 있다면 다른 은유를 사용하거나 다른 작업 패턴을 권장해 보세요. 시도하는 비용이 비교적 저렴하며 훌륭한 것을 찾을 수도 있습니다.

2. **프롬프트를 간결하게 유지합니다.** 역할을 하지 않는 것들을 제거합니다. 최종 출력만이 아니라 반드시 전사를 읽으세요 — 스킬이 모델로 하여금 비생산적인 일을 하는 데 많은 시간을 낭비하게 만드는 것처럼 보이면, 그렇게 만드는 스킬 부분을 제거하고 어떻게 되는지 확인해 볼 수 있습니다.

3. **이유를 설명합니다.** 요청하는 모든 것의 **이유**를 설명하려고 열심히 노력합니다. 오늘날의 LLM은 *똑똑합니다*. 좋은 마음의 이론을 가지고 있으며 좋은 도구가 주어지면 단순한 지시를 넘어서 정말 일을 해낼 수 있습니다. 사용자의 피드백이 퉁명스럽거나 좌절스러워 보이더라도, 실제로 작업을 이해하려고 노력하고, 사용자가 왜 그것을 썼는지, 실제로 무엇을 원하는지 이해하고, 이 이해를 지시사항에 전달합니다. 모두 대문자로 ALWAYS 또는 NEVER를 쓰거나, 매우 엄격한 구조를 사용하는 것은 노란 신호입니다 — 가능하면 재구성하고 요청하는 것이 왜 중요한지 이유를 설명합니다. 그것이 더 인간적이고, 강력하며, 효과적인 접근법입니다.

4. **여러 테스트 케이스에 걸친 반복 작업을 찾습니다.** 테스트 실행의 전사를 읽고 서브에이전트들이 모두 독립적으로 유사한 헬퍼 스크립트를 작성하거나 무언가에 대해 동일한 다단계 접근법을 취했는지 확인합니다. 3개의 테스트 케이스 모두 서브에이전트가 `create_docx.py`나 `build_chart.py`를 작성하는 결과를 낳았다면, 그것은 스킬이 그 스크립트를 번들해야 한다는 강력한 신호입니다. 한 번 작성하고, `scripts/`에 넣고, 스킬이 그것을 사용하도록 알려줍니다. 이렇게 하면 미래의 모든 호출이 바퀴를 다시 발명하는 것을 방지합니다.

이 작업은 상당히 중요하며 (여기서 연간 수십억 달러의 경제적 가치를 창출하려고 노력합니다!) 당신의 생각하는 시간이 병목이 아닙니다. 시간을 갖고 정말로 깊이 생각하세요. 초안 개정을 작성한 다음 새로운 시각으로 보고 개선하는 것을 제안합니다. 사용자의 머릿속에 들어가 정말로 원하는 것이 무엇인지 이해하는 데 최선을 다합니다.

### 반복 루프

스킬을 개선한 후:

1. 스킬에 개선 사항 적용
2. 새 `iteration-<N+1>/` 디렉토리에 모든 테스트 케이스를 다시 실행합니다 (baseline 실행 포함). 새 스킬을 만드는 경우, baseline은 항상 `without_skill` (스킬 없음) — 반복 전반에 걸쳐 동일하게 유지됩니다. 기존 스킬을 개선하는 경우, baseline으로 무엇이 의미 있는지 판단합니다: 사용자가 가져온 원본 버전 또는 이전 반복.
3. 이전 반복을 가리키는 `--previous-workspace`와 함께 리뷰어를 실행합니다
4. 사용자가 검토하고 완료되었다고 알릴 때까지 기다립니다
5. 새 피드백을 읽고, 다시 개선하고, 반복합니다

다음 중 하나가 될 때까지 계속합니다:
- 사용자가 만족한다고 말할 때
- 피드백이 모두 비어 있을 때 (모든 것이 좋아 보임)
- 의미 있는 진전을 이루지 못할 때

## 고급: 블라인드 비교

두 버전의 스킬을 더 엄격하게 비교하고 싶은 상황에서 (예: 사용자가 "새 버전이 실제로 더 낫나요?"라고 물을 때) 블라인드 비교 시스템이 있습니다. 자세한 내용은 `agents/comparator.md`와 `agents/analyzer.md`를 읽으세요. 기본 개념은: 두 출력을 어떤 것이 어떤 것인지 알리지 않고 독립적인 에이전트에게 주고 품질을 판단하게 합니다. 그런 다음 왜 승자가 이겼는지 분석합니다.

이것은 선택적이며, 서브에이전트가 필요하고, 대부분의 사용자는 필요하지 않을 것입니다. 인간 리뷰 루프가 보통 충분합니다.

## Description 최적화

SKILL.md frontmatter의 description 필드는 Claude가 스킬을 호출할지 결정하는 주요 메커니즘입니다. 스킬을 만들거나 개선한 후, 더 나은 트리거 정확도를 위해 description을 최적화할 것을 제안합니다.

### 1단계: 트리거 eval 쿼리 생성

20개의 eval 쿼리를 만듭니다 — 트리거해야 하는 것과 트리거하지 않아야 하는 것의 혼합. JSON으로 저장합니다:

```json
[
  {"query": "사용자 프롬프트", "should_trigger": true},
  {"query": "다른 프롬프트", "should_trigger": false}
]
```

쿼리는 현실적이며 Claude Code 또는 Claude.ai 사용자가 실제로 입력할 내용이어야 합니다. 추상적인 요청이 아닌, 구체적이고 상세한 요청. 예를 들어, 파일 경로, 사용자의 직업이나 상황에 대한 개인적인 컨텍스트, 열 이름과 값, 회사 이름, URL. 약간의 배경 스토리. 일부는 소문자이거나 약어, 오타, 또는 구어체가 포함될 수 있습니다. 다양한 길이를 사용하고, 명확한 것보다 엣지 케이스에 집중합니다 (사용자는 검토 및 승인 기회가 있을 것).

나쁨: `"이 데이터 형식 지정"`, `"PDF에서 텍스트 추출"`, `"차트 만들기"`

좋음: `"ok 상사가 방금 xlsx 파일을 보냈는데 (다운로드에 있음, 'Q4 sales final FINAL v2.xlsx' 같은 이름인 것 같음) 수익 대비 이익 마진을 백분율로 보여주는 열을 추가하고 싶대요. 수익은 C열에 있고 비용은 D열인 것 같아요"`

**트리거해야 하는 쿼리 (8-10개)**: 커버리지에 대해 생각합니다. 동일한 의도의 다른 표현이 필요합니다 — 일부는 공식적이고, 일부는 비공식적. 사용자가 스킬이나 파일 유형을 명시적으로 명시하지 않지만 분명히 필요한 경우를 포함합니다. 드문 사용 사례와 이 스킬이 다른 것과 경쟁하지만 이겨야 하는 경우를 추가합니다.

**트리거하지 않아야 하는 쿼리 (8-10개)**: 가장 가치 있는 것은 근접 실패 — 스킬과 키워드나 개념을 공유하지만 실제로는 다른 것이 필요한 쿼리. 인접 도메인, 나이브한 키워드 매치는 트리거하지만 트리거하지 않아야 하는 모호한 표현, 스킬이 하는 것과 관련있지만 다른 도구가 더 적합한 컨텍스트인 경우를 생각합니다.

피해야 할 주요 사항: 트리거하지 않아야 하는 쿼리를 명백히 관련 없게 만들지 마세요. PDF 스킬에 대한 부정적 테스트로 "피보나치 함수 작성"은 너무 쉽습니다 — 아무것도 테스트하지 않습니다. 부정적인 경우는 진정으로 까다로워야 합니다.

### 2단계: 사용자와 검토

HTML 템플릿을 사용하여 eval 세트를 사용자에게 검토용으로 제시합니다:

1. `assets/eval_review.html`에서 템플릿을 읽습니다
2. 플레이스홀더를 교체합니다:
   - `__EVAL_DATA_PLACEHOLDER__` → eval 항목의 JSON 배열 (따옴표 없이 — JS 변수 할당)
   - `__SKILL_NAME_PLACEHOLDER__` → 스킬 이름
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 스킬의 현재 description
3. 임시 파일에 씁니다 (예: `/tmp/eval_review_<skill-name>.html`) 그리고 엽니다: `open /tmp/eval_review_<skill-name>.html`
4. 사용자는 쿼리를 편집하고, should-trigger를 토글하고, 항목을 추가/제거한 다음 "Eval 세트 내보내기" 클릭 가능
5. 파일은 `~/Downloads/eval_set.json`에 다운로드됩니다 — 여러 개가 있는 경우 가장 최근 버전을 확인합니다 (예: `eval_set (1).json`)

이 단계가 중요합니다 — 나쁜 eval 쿼리는 나쁜 description으로 이어집니다.

### 3단계: 최적화 루프 실행

사용자에게 알립니다: "이 작업은 시간이 걸립니다 — 백그라운드에서 최적화 루프를 실행하고 주기적으로 확인하겠습니다."

eval 세트를 workspace에 저장한 다음 백그라운드에서 실행합니다:

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

현재 세션을 구동하는 시스템 프롬프트의 모델 ID를 사용합니다. 이렇게 하면 트리거 테스트가 사용자가 실제로 경험하는 것과 일치합니다.

실행 중에 주기적으로 출력을 tail하여 어떤 반복에 있는지와 점수가 어떻게 되는지 업데이트를 제공합니다.

이것은 전체 최적화 루프를 자동으로 처리합니다. eval 세트를 60% 훈련과 40% 보류 테스트로 분할하고, 현재 description을 평가하며 (신뢰할 수 있는 트리거율을 얻기 위해 각 쿼리를 3번 실행), 실패한 것을 기반으로 개선을 제안하기 위해 확장된 사고로 Claude를 호출합니다. 훈련과 테스트 모두에서 각 새 description을 재평가하며 최대 5번 반복합니다. 완료되면 브라우저에서 결과를 보여주는 HTML 보고서를 열고 `best_description`이 있는 JSON을 반환합니다 — 과적합을 피하기 위해 훈련 점수가 아닌 테스트 점수로 선택됩니다.

### 스킬 트리거 작동 방식

트리거 메커니즘을 이해하면 더 나은 eval 쿼리를 설계하는 데 도움이 됩니다. 스킬은 Claude의 `available_skills` 목록에 이름 + description과 함께 나타나며, Claude는 해당 description을 기반으로 스킬을 참조할지 여부를 결정합니다. 중요한 것은 Claude는 스스로 쉽게 처리할 수 없는 작업에만 스킬을 참조한다는 것입니다 — "이 PDF 읽기"와 같은 간단한 단일 단계 쿼리는 description이 완벽하게 일치하더라도 스킬을 트리거하지 않을 수 있습니다. 복잡하고 다단계이며 전문화된 쿼리는 description이 일치할 때 스킬을 안정적으로 트리거합니다.

이는 eval 쿼리가 Claude가 실제로 스킬을 참조하는 것으로 이익을 얻을 만큼 실질적이어야 함을 의미합니다. "파일 X 읽기"와 같은 간단한 쿼리는 description 품질에 관계없이 스킬을 트리거하지 않으므로 좋지 않은 테스트 케이스입니다.

### 4단계: 결과 적용

JSON 출력에서 `best_description`을 가져와 스킬의 SKILL.md frontmatter를 업데이트합니다. 변경 전/후를 사용자에게 보여주고 점수를 보고합니다.

## Claude.ai 전용 지시사항

Claude.ai에서 핵심 워크플로는 동일하지만 (초안 → 테스트 → 검토 → 개선 → 반복), Claude.ai에는 서브에이전트가 없으므로 일부 메커니즘이 변경됩니다. 변경할 사항은 다음과 같습니다:

**테스트 케이스 실행**: 서브에이전트가 없으므로 병렬 실행이 불가능합니다. 각 테스트 케이스에 대해 스킬의 SKILL.md를 읽은 다음 지시사항을 따라 스스로 테스트 프롬프트를 달성합니다. 하나씩 처리합니다. 이는 독립적인 서브에이전트보다 덜 엄격하지만 (스킬을 작성하고 실행하므로 전체 컨텍스트가 있음), 유용한 합리성 검사입니다 — 인간 검토 단계가 보완합니다. Baseline 실행은 건너뜁니다 — 요청된 대로 스킬을 사용하여 작업을 완료합니다.

**결과 검토**: 브라우저를 열 수 없는 경우 (예: Claude.ai의 VM에 디스플레이가 없거나 원격 서버에 있는 경우), 브라우저 리뷰어를 완전히 건너뜁니다. 대신 결과를 대화에서 직접 제시합니다. 각 테스트 케이스에 대해 프롬프트와 출력을 보여줍니다. 출력이 사용자가 봐야 하는 파일인 경우 (예: .docx 또는 .xlsx), 파일 시스템에 저장하고 다운로드하여 검사할 수 있는 위치를 알려줍니다. 인라인으로 피드백을 요청합니다: "이것이 어떻게 보이나요? 변경하고 싶은 것이 있나요?"

**벤치마킹**: 정량적 벤치마킹을 건너뜁니다 — 서브에이전트 없이는 의미 있는 baseline 비교에 의존합니다. 사용자의 정성적 피드백에 집중합니다.

**반복 루프**: 이전과 동일합니다 — 스킬을 개선하고, 테스트 케이스를 다시 실행하고, 피드백을 요청합니다 — 브라우저 리뷰어 없이. 파일 시스템이 있는 경우 반복 디렉토리에 결과를 구성할 수 있습니다.

**Description 최적화**: 이 섹션은 Claude Code에서만 사용 가능한 `claude` CLI 도구 (특히 `claude -p`)가 필요합니다. Claude.ai에서는 건너뜁니다.

**블라인드 비교**: 서브에이전트가 필요합니다. 건너뜁니다.

**패키징**: `package_skill.py` 스크립트는 Python과 파일 시스템이 있는 어디서나 작동합니다. Claude.ai에서 실행하고 사용자는 결과 `.skill` 파일을 다운로드할 수 있습니다.

## Cowork 전용 지시사항

Cowork에 있는 경우 알아야 할 주요 사항:

- 서브에이전트가 있으므로 메인 워크플로 (병렬로 테스트 케이스 스폰, baseline 실행, 채점 등)가 모두 작동합니다. (하지만 타임아웃에 심각한 문제가 있으면 병렬 대신 순차적으로 테스트 프롬프트를 실행해도 됩니다.)
- 브라우저나 디스플레이가 없으므로 eval 뷰어를 생성할 때 서버를 시작하는 대신 `--static <output_path>`를 사용하여 독립형 HTML 파일을 작성합니다. 그런 다음 사용자가 브라우저에서 HTML을 열 수 있는 링크를 제공합니다.
- 어떤 이유로 Cowork 설정이 테스트를 실행한 후 eval 뷰어를 생성하지 않도록 Claude를 설득하는 것 같으므로, Cowork에서든 Claude Code에서든 테스트 실행 후 스킬을 직접 수정하고 수정하려고 하기 전에 인간이 예시를 검토할 수 있도록 eval 뷰어를 항상 생성해야 합니다. `generate_review.py`를 사용합니다 (자체적인 맞춤 HTML 코드를 작성하지 마세요). 미리 사과드립니다만 여기서는 대문자로 쓰겠습니다: 입력을 직접 평가하기 전에 EVAL VIEWER를 *먼저* 생성합니다. 가능한 한 빨리 인간에게 보여주고 싶습니다!
- 피드백 작동 방식이 다릅니다: 실행 중인 서버가 없으므로 뷰어의 "모든 리뷰 제출" 버튼이 `feedback.json`을 파일로 다운로드합니다. 그런 다음 거기서 읽을 수 있습니다 (먼저 접근 권한을 요청해야 할 수 있음).
- 패키징이 작동합니다 — `package_skill.py`는 Python과 파일 시스템만 필요합니다.
- Description 최적화 (`run_loop.py` / `run_eval.py`)는 브라우저가 아닌 subprocess를 통해 `claude -p`를 사용하므로 Cowork에서 잘 작동해야 하지만, 스킬이 완전히 완성되고 사용자가 좋은 상태라는 데 동의할 때까지 저장하세요.
