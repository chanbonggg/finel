# Frontend Slides

브라우저에서 완전히 실행되는 의존성 없는 애니메이션이 풍부한 HTML 프레젠테이션 제작.

zarazhangrui의 작업에서 보여준 시각적 탐색 접근 방식에서 영감을 받음 (크레딧: @zarazhangrui).

## 활성화 시점

- 발표용, 피치용, 워크숍용 또는 내부 프레젠테이션 덱 제작
- `.ppt` 또는 `.pptx` 슬라이드를 HTML 프레젠테이션으로 변환
- 기존 HTML 프레젠테이션의 레이아웃, 모션 또는 타이포그래피 개선
- 디자인 선호도를 모르는 사용자와 함께 프레젠테이션 스타일 탐색

## 필수 조건

1. **의존성 없음**: 기본적으로 인라인 CSS와 JS를 포함한 하나의 독립적인 HTML 파일.
2. **뷰포트 맞춤은 필수**: 모든 슬라이드는 내부 스크롤 없이 하나의 뷰포트 내에 맞아야 함.
3. **보여주기, 말하지 않기**: 추상적인 스타일 설문 대신 시각적 미리보기 사용.
4. **독특한 디자인**: 일반적인 보라색 그라디언트, 흰 배경에 Inter, 템플릿 같은 덱 피하기.
5. **프로덕션 품질**: 코드에 주석을 달고, 접근 가능하고, 반응형이며, 성능이 좋게 유지.

생성 전에 뷰포트 안전 CSS 베이스, 밀도 제한, 프리셋 카탈로그 및 CSS 주의사항을 위해 `STYLE_PRESETS.md`를 읽으세요.

## 워크플로

### 1. 모드 감지

하나의 경로를 선택:
- **새 프레젠테이션**: 사용자에게 주제, 노트 또는 전체 초안이 있는 경우
- **PPT 변환**: 사용자에게 `.ppt` 또는 `.pptx`가 있는 경우
- **개선**: 사용자가 이미 HTML 슬라이드를 갖고 있으며 개선을 원하는 경우

### 2. 콘텐츠 파악

필요한 최소한만 질문:
- 목적: 피치, 교육, 컨퍼런스 발표, 내부 업데이트
- 길이: 짧음 (5-10), 중간 (10-20), 긴 것 (20+)
- 콘텐츠 상태: 완성된 텍스트, 거친 노트, 주제만

사용자에게 콘텐츠가 있으면 스타일링 전에 붙여넣도록 요청.

### 3. 스타일 파악

기본적으로 시각적 탐색 방식을 사용.

사용자가 원하는 프리셋을 이미 알고 있으면 미리보기를 건너뛰고 바로 사용.

그렇지 않으면:
1. 덱이 어떤 느낌을 만들어야 하는지 질문: 감동, 활력, 집중, 영감.
2. `.ecc-design/slide-previews/`에 **3개의 단일 슬라이드 미리보기 파일** 생성.
3. 각 미리보기는 독립적이어야 하고, 타이포그래피/색상/모션을 명확히 보여주며, 슬라이드 콘텐츠 약 100줄 이하로 유지.
4. 사용자에게 유지할 미리보기 또는 혼합할 요소를 질문.

분위기를 스타일로 매핑할 때 `STYLE_PRESETS.md`의 프리셋 가이드 사용.

### 4. 프레젠테이션 빌드

출력:
- `presentation.html`
- `[presentation-name].html`

덱에 추출된 또는 사용자가 제공한 이미지가 포함된 경우에만 `assets/` 폴더 사용.

필수 구조:
- 시맨틱 슬라이드 섹션
- `STYLE_PRESETS.md`의 뷰포트 안전 CSS 베이스
- 테마 값을 위한 CSS 커스텀 프로퍼티
- 키보드, 휠, 터치 네비게이션을 위한 프레젠테이션 컨트롤러 클래스
- 공개 애니메이션을 위한 Intersection Observer
- 모션 감소 지원

### 5. 뷰포트 맞춤 강제

이것을 필수 게이트로 취급.

규칙:
- 모든 `.slide`는 `height: 100vh; height: 100dvh; overflow: hidden;` 사용 필수
- 모든 타입과 간격은 `clamp()`로 스케일
- 콘텐츠가 맞지 않으면 여러 슬라이드로 분할
- 읽을 수 없는 크기로 텍스트를 축소하여 오버플로 해결 금지
- 슬라이드 내부의 스크롤바 금지

`STYLE_PRESETS.md`의 밀도 제한 및 필수 CSS 블록 사용.

### 6. 검증

다음 크기에서 완성된 덱 확인:
- 1920x1080
- 1280x720
- 768x1024
- 375x667
- 667x375

브라우저 자동화를 사용할 수 있으면, 슬라이드 오버플로 없음과 키보드 네비게이션 작동 여부 확인에 사용.

### 7. 전달

인계 시:
- 사용자가 유지하길 원하지 않으면 임시 미리보기 파일 삭제
- 유용한 경우 플랫폼에 적합한 오프너로 덱 열기
- 파일 경로, 사용한 프리셋, 슬라이드 수, 쉬운 테마 커스터마이징 포인트 요약

현재 OS에 맞는 올바른 오프너 사용:
- macOS: `open file.html`
- Linux: `xdg-open file.html`
- Windows: `start "" file.html`

## PPT / PPTX 변환

PowerPoint 변환의 경우:
1. 텍스트, 이미지, 노트 추출에 `python-pptx`를 사용하는 `python3` 선호.
2. `python-pptx`를 사용할 수 없으면 설치 여부를 묻거나 수동/내보내기 기반 워크플로로 대체.
3. 슬라이드 순서, 발표자 노트 및 추출된 에셋 보존.
4. 추출 후 새 프레젠테이션과 동일한 스타일 선택 워크플로 실행.

변환을 크로스플랫폼으로 유지. Python이 처리할 수 있을 때 macOS 전용 도구에 의존 금지.

## 구현 요구사항

### HTML / CSS

- 사용자가 명시적으로 다중 파일 프로젝트를 원하지 않으면 인라인 CSS와 JS 사용.
- 폰트는 Google Fonts 또는 Fontshare에서 가져올 수 있음.
- 분위기 있는 배경, 강한 타입 계층, 명확한 시각적 방향 선호.
- 일러스트레이션보다 추상적인 형태, 그라디언트, 그리드, 노이즈, 기하학 사용.

### JavaScript

포함 항목:
- 키보드 네비게이션
- 터치/스와이프 네비게이션
- 마우스 휠 네비게이션
- 진행 표시기 또는 슬라이드 인덱스
- 진입 시 공개 애니메이션 트리거

### 접근성

- 시맨틱 구조 사용 (`main`, `section`, `nav`)
- 대비를 읽을 수 있게 유지
- 키보드 전용 네비게이션 지원
- `prefers-reduced-motion` 존중

## 콘텐츠 밀도 제한

사용자가 더 높은 밀도를 명시적으로 요청하고 가독성이 여전히 유지되지 않는 한 이 최대값 사용:

| 슬라이드 유형 | 제한 |
|------------|-------|
| 제목 | 1 제목 + 1 부제목 + 선택적 태그라인 |
| 콘텐츠 | 1 제목 + 4-6 불릿 또는 2 짧은 단락 |
| 기능 그리드 | 최대 6개 카드 |
| 코드 | 최대 8-10줄 |
| 인용 | 1 인용 + 출처 |
| 이미지 | 뷰포트에 맞게 제한된 1 이미지 |

## 안티패턴

- 시각적 정체성이 없는 일반적인 스타트업 그라디언트
- 의도적으로 에디토리얼하지 않은 이상 시스템 폰트 덱
- 긴 불릿 목록
- 스크롤이 필요한 코드 블록
- 짧은 화면에서 깨지는 고정 높이 콘텐츠 박스
- `-clamp(...)`와 같은 잘못된 부정된 CSS 함수

## 납품 체크리스트

- 브라우저에서 로컬 파일로 프레젠테이션 실행 가능
- 모든 슬라이드가 스크롤 없이 뷰포트에 맞음
- 스타일이 독특하고 의도적
- 애니메이션이 의미 있고 소음 없음
- 모션 감소가 존중됨
- 인계 시 파일 경로와 커스터마이징 포인트 설명
