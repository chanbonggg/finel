name: commit
description: git 변경 사항을 분석해서 한국어 커밋 메시지를 작성하고 커밋 후 푸시까지 진행한다
---

git status, git diff를 확인한 후 변경 사항을 분석해서 한국어 커밋 메시지를 작성하고 커밋 후 푸시까지 진행한다.

## 실행 순서

### 1단계 — 현재 상태 파악

다음 명령어를 실행해서 현재 상태를 확인한다:

```bash
git status
git diff
git diff --staged
git log --oneline -5
```

### 2단계 — 변경 사항 분석

파악한 내용을 바탕으로:
- 어떤 파일이 변경/추가/삭제됐는지 확인
- 변경의 성격 파악 (기능 추가 / 버그 수정 / 리팩터링 / 설정 변경 / 문서 등)
- 핵심 변경 내용 요약

### 3단계 — 커밋 메시지 작성 규칙

**제목 형식:** `[타입] 한국어 설명`

타입 분류:
- `[기능]` — 새로운 기능 추가
- `[수정]` — 버그 수정
- `[개선]` — 기존 기능 개선
- `[리팩터]` — 코드 구조 개선 (동작 변경 없음)
- `[설정]` — 환경설정, 패키지, 빌드 관련
- `[삭제]` — 파일 또는 코드 제거
- `[문서]` — 주석, README 등 문서 변경
- `[스타일]` — UI/CSS 변경

**제목:** 50자 이내, 명사형 또는 동사형으로 끝내기
**본문:** 변경 파일이 3개 이상이거나 복잡한 경우 본문에 상세 내용 추가

예시:
```
[기능] 상품 상세 페이지 관련 상품 추천 기능 추가
[수정] 로그인 토큰 만료 시 리다이렉트 오류 수정
[설정] Cloudinary 이미지 업로드 환경변수 추가
```

### 4단계 — 스테이징

`$ARGUMENTS`가 있으면 그 내용을 커밋 메시지 힌트로 활용한다.

**파일 분류 기준:**

1. `git status`로 다음 3가지 상태를 모두 확인한다:
   - Untracked files (새 파일, 아직 한 번도 add 안 됨)
   - Changes not staged (수정됐지만 add 안 됨)
   - Changes to be committed (이미 staged)

2. 아래 파일은 절대 add하지 않는다:
   - `.env`, `.env.local`, `.env.*`
   - `*secret*`, `*credential*`, `*token*`
   - `node_modules/`
   - `.claude/` 디렉터리 (에이전트/스킬 파일 포함)

3. 스테이징 안 된 파일(untracked + unstaged 수정)이 있으면 **자동으로 전부 add**한다:

```bash
git add -A
```

단, 민감 파일이 섞여 있으면 파일별로 골라서 add한다:

```bash
git add [안전한 파일1] [안전한 파일2] ...
```

### 5단계 — 커밋

커밋 메시지는 HEREDOC 방식으로 전달한다:

```bash
git commit -m "$(cat <<'EOF'
[타입] 한국어 커밋 제목

- 변경 내용 1
- 변경 내용 2
EOF
)"
```

### 6단계 — 푸시

현재 브랜치를 확인하고 푸시한다:

```bash
git branch --show-current
git push origin [현재 브랜치]
```

- 처음 푸시하는 브랜치라면 `-u` 플래그를 추가한다
- `main` 또는 `master` 브랜치에 직접 푸시하는 경우 사용자에게 한번 확인한다

### 7단계 — 완료 보고

다음 형식으로 결과를 출력한다:

```
## 커밋 완료

- 브랜치: [브랜치명]
- 커밋 해시: [hash]
- 커밋 메시지: [작성한 메시지]
- 변경 파일: N개
- 푸시: 완료
```
