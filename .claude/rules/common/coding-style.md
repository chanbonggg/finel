# 코딩 스타일

## 불변성 (중요)

항상 새 객체를 생성하고, 기존 객체를 절대 변경하지 않습니다:

```
// 의사 코드
잘못됨:  modify(original, field, value) → original을 직접 변경
올바름: update(original, field, value) → 변경된 새 복사본 반환
```

근거: 불변 데이터는 숨겨진 사이드 이펙트를 방지하고, 디버깅을 쉽게 만들며, 안전한 동시성을 가능하게 합니다.

## 파일 구성

많은 작은 파일 > 적은 큰 파일:
- 높은 응집도, 낮은 결합도
- 일반적으로 200-400줄, 최대 800줄
- 큰 모듈에서 유틸리티 추출
- 타입이 아닌 기능/도메인 기준으로 구성

## 에러 처리

항상 에러를 포괄적으로 처리합니다:
- 모든 계층에서 명시적으로 에러 처리
- UI 노출 코드에서는 사용자 친화적인 에러 메시지 제공
- 서버 사이드에서는 상세한 에러 컨텍스트 로깅
- 에러를 조용히 삼키지 않습니다

## 입력 유효성 검사

항상 시스템 경계에서 유효성을 검사합니다:
- 처리 전 모든 사용자 입력 유효성 검사
- 가능한 경우 스키마 기반 유효성 검사 사용
- 명확한 에러 메시지와 함께 빠르게 실패
- 외부 데이터(API 응답, 사용자 입력, 파일 내용)를 절대 신뢰하지 않습니다

## 코드 품질 체크리스트

작업 완료 전 확인:
- [ ] 코드가 읽기 쉽고 이름이 명확함
- [ ] 함수가 작음 (50줄 미만)
- [ ] 파일이 집중적임 (800줄 미만)
- [ ] 깊은 중첩 없음 (4단계 초과)
- [ ] 적절한 에러 처리
- [ ] 하드코딩된 값 없음 (상수 또는 config 사용)
- [ ] 변경 없음 (불변 패턴 사용)
