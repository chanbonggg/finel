# 공통 패턴

## 스켈레톤 프로젝트

새 기능을 구현할 때:
1. 검증된 스켈레톤 프로젝트 검색
2. 병렬 agents를 사용하여 옵션 평가:
   - 보안 평가
   - 확장성 분석
   - 관련성 점수
   - 구현 계획
3. 최적의 후보를 기반으로 클론
4. 검증된 구조 내에서 반복 개발

## 디자인 패턴

### Repository 패턴

일관된 인터페이스 뒤에 데이터 접근을 캡슐화합니다:
- 표준 작업 정의: findAll, findById, create, update, delete
- 구체적인 구현이 스토리지 세부사항 처리 (데이터베이스, API, 파일 등)
- 비즈니스 로직은 스토리지 메커니즘이 아닌 추상 인터페이스에 의존
- 데이터 소스 교체를 쉽게 하고 mock으로 테스트 단순화

### API 응답 형식

모든 API 응답에 일관된 봉투(envelope) 사용:
- 성공/상태 표시자 포함
- 데이터 페이로드 포함 (에러 시 null 허용)
- 에러 메시지 필드 포함 (성공 시 null 허용)
- 페이지네이션 응답에 메타데이터 포함 (total, page, limit)
